# 개발로그 쓰는 스킬을 만들었다 — 근데 '개발로그'부터 잘못 알고 있었다

> 공부하고 삽질한 걸 티스토리에 올리고 싶은데, 매번 손으로 글을 다듬는 게 일이었다. 그래서 그걸 대신 해주는 클로드 스킬을 만들었다. 그리고 첫 설계부터 틀렸다.

지금 읽는 이 글은 **그 스킬이 스스로에 대해 쓴 글**이다. 사람이 뼈대를 잡은 게 아니라, "이 스킬 만든 이야기 개발로그로 써줘" 한마디로 나왔다.

---

## 결과물 먼저 — 뭘 만들었나

`devlog` 스킬. 두 가지를 한다.

- **기록** — 문제·삽질·깨달음을 주제별 로그로 쌓는다. 실패한 시도도 그대로 남긴다. 나중에 글의 재료가 되니까.
- **발행** — 쌓인 로그(또는 방금 한 작업)를 티스토리에 바로 올릴 개발로그 글로 정리한다.

```
skills/devlog/
├── SKILL.md                 # 기록 + 발행 두 모드
└── references/
    ├── blog-craft.md        # 잘 읽히는 글쓰기 원칙
    └── category-guide.md    # 티스토리 카테고리 가이드
```

원래 `learning-notes`라는 개인 노트 스킬이 있었는데, 그건 "미래의 내가 다시 보려고" 남기는 거였다. 발행용이 아니었다. 그래서 개발로그로 갈아엎고 발행 기능을 붙였다.

---

## 삽질 1 — '개발로그'를 반쪽만 알고 있었다

개발로그가 뭐냐고 하면 나는 **삽질기**라고 생각했다. 문제 만나고, 헤매고, 풀고, 배운 것. 그래서 발행 구조를 이렇게 못박았다.

```
문제 → 삽질 → 깨달음 → 결과 → 다음
```

`blog-craft.md`도 이 흐름 위주로 썼다. 실패 루트를 분리해라, 구체적 실패를 묘사해라, 검증 섹션을 둬라. 다 삽질기 이야기였다.

그런데 스킬을 다 만들고 나서, 잘 쓰인 글 하나를 예시로 받았다. **"브라우저는 미니 OS다 — 저장 영역 전부 정리"** 같은 글이었다.

읽어보니 삽질이 한 줄도 없었다. 문제도 실패도 없었다. 그냥 **뭔가를 이해하고, 그 이해를 남이 이해하게 정리한 글**이었다. 비교표가 있고, "sessionStorage는 서버 세션이 아니다" 같은 헷갈리는 점을 콕 집어주고, "임차 메모리" 같은 비유가 있었다.

여기서 소거가 됐다. 내 스킬을 그대로 이 글에 적용했으면? **개념 정리 글에 억지로 "삽질" 틀을 씌웠을 거다.** 있지도 않은 실패 에피소드를 지어내라고 압박했을 거다. 최악이다.

깨달음: **개발로그는 두 결이다.**

| | 개념 정리형 | 삽질기형 |
|---|---|---|
| 언제 | 뭘 **이해**했을 때 | 뭘 **만들며 겪었**을 때 |
| 뼈대 | 훅 → 개념·비교표 → 헷갈리는 점 → 비유 → 정리 | 결과물 먼저 → 문제 → 삽질 → 해결 → 검증 |
| 재료 | 비유·비교표·헷갈리는 점 | 실패한 시도·정량 증거 |

그래서 `blog-craft.md`를 다시 썼다. 발행 첫 단계를 **"이 글이 어느 결인지 먼저 정한다"**로 바꿨다. 내용이 아키타입을 정한다. 억지로 한 틀에 안 씌운다.

---

## 삽질 2 — 카테고리와 글의 결을 헷갈릴 뻔했다

다음은 티스토리 카테고리. 처음엔 "개발로그 카테고리 / 개념 정리 카테고리"로 나누려 했다. 글의 성격으로 카테고리를 나누는 거다.

참고할 블로그를 하나 봤다. 카테고리가 이랬다.

```
회고 · 서버 · 디비 · 자바 · 자바스크립트 · 학습법 · 스타트업
```

(이미지: 참고 블로그 youngsuk-dev 의 카테고리 목록 — 전부 '주제'다)

전부 **주제**였다. 글이 삽질기냐 개념 정리냐로 나눈 게 아니라, 무엇에 관한 글이냐로 나눠져 있었다.

여기서 두 번째 깨달음. **카테고리(주제)와 아키타입(글쓰기 결)은 직교한다.**

- "브라우저 저장소 정리" = CS 카테고리 + 개념 정리형
- "디자인 스킬 만들기" = AI·클로드 카테고리 + 삽질기형

같은 카테고리 안에 두 결이 섞여도 정상이다. 둘을 한 축으로 묶으려던 게 실수였다. 그래서 카테고리는 주제로만 3개 뒀다 — AI·클로드 / CS·프론트엔드 / 회고. 글의 결은 발행할 때 내용 보고 정한다.

---

## 검증 — 이 글이 증거다

말로만 "두 아키타입 된다"고 하면 안 믿긴다. 그래서 직접 돌려봤다.

바로 앞에 만든 디자인 스킬 이야기는 **삽질기형**으로 뽑혔다 (린터 버그 잡은 이야기, 결과물 먼저). 그리고 지금 이 글은 같은 스킬로 뽑은 **또 다른 삽질기형**이다. 개념 정리형이 필요하면 브라우저 저장소 글처럼 훅·비교표·비유로 나온다.

한 스킬이 내용에 따라 구조를 바꾼다. 이게 첫 설계(삽질기 고정)와 지금의 차이다.

---

## 정리

- 개발로그는 **삽질기만이 아니다.** 뭘 이해한 글(개념 정리형)과 뭘 겪은 글(삽질기형), 두 결이다. 내용이 결을 정한다.
- **카테고리(주제) ⟂ 아키타입(글쓰기 결).** 한 축으로 묶으면 안 된다.
- 스킬에 "까먹으면 안 되는 개념"을 파일로 박아두면(blog-craft·category-guide), 매번 그 원칙대로 나온다. 내 기억력에 안 맡긴다.
