## 배경

v2.3.x 패치를 거치면서 두 가지 사용자 피로 신호가 누적됐습니다.

1. **안내 톤이 너무 어렵다** — 사용자가 직접 신고했습니다. 영어 식별자가 많고, 한 줄에 정보를 압축하고, 메모 표기 (`→`, `OK`, 콜론 다발) 가 자주 쓰여서 읽기 어렵다는 의견입니다. 사용자 인용:

   > "그리고 전체적으로 그 안내하는 말들이 너무 어렵게 말하는경향이 있어.."

2. **비동기 .html 생성이 가끔 실패한다** — 요구사항/기술설계/구현계획 처음 작성 시 백그라운드로 생성하는 `.html` 동봉본이 가끔 안 만들어집니다. 사용자 인용:

   > "그리고 비동기로 html 생성이 안되는경우가 있어 (처음 md 생성시) 이것도 2.4v에 같이 넣어줘"

두 항목 모두 사용자 직격이라 같이 묶어서 메이저 한 번에 처리합니다.

| A-4 사용자 친화 보고 | 무엇을 했는지 (1~2 문장) + 왜 그랬는지 (필요 시) + 다음 단계 (1 문장). 그 외 세부는 자세히 묻기 전 생략. |
| A-5 영역별 적용 구분 | 위 D2 표 그대로. |

## 우려 / 해결

### Q1 — A 톤 변경이 skill 동작에 영향?

영향 0. A 는 사용자 노출 문구만 한국어 친화. skill body 의 룰 본문 / 코드 패턴은 영어 그대로 (메인 prompt 가공용).

### Q2 — A 적용 후 새 합류자 / 영어 환경 사용자 영향?

skill body 의 영어 룰 본문은 그대로. 사용자 노출 문구만 한국어로 변경. 영어 환경 사용자는 영어 식별자 / 도구 이름은 유지되므로 코드 작업 지장 없음. 안내문만 한국어.

### Q3 — B 의 sync fallback 자동 retry 미도입 위험?

사용자 명시 호출만 trigger → 사용자가 `.html` 부재 catch 못 하면 영구 stale. mitigation: 메인 응답에 dispatch 결과 명시 (B-4) + silent log (B-3). 사용자가 transition notice 시점에 확인 가능.

### Q4 — D6 wave 분해 — wave 7 다 끝나야 release? 부분 release?

D6 wave 끝까지 단일 v2.4.0. 부분 release 안 함. 백로그 권장 답습.

### Q5 — A 톤 변경이 og-* skill 영향?

영향 0. og-* 는 upstream mirror 보존 (변경 X).

### Q6 — B race condition delay 5초가 사용자 latency?

영향 미미. 메인 turn 의 다음 step 진행에 5초 추가. 사용자가 응답 기다리는 시점 아님 (자동 chain 안). transition notice 시점 도착 전.

### Q7 — A 적용 검증 시점 — release 후?

Wave 단위 commit 직후 사용자 spot-check 옵션 있음. release 전 dogfood. 메인이 wave 끝에서 sample 1~2개 (변경 전 / 변경 후) 사용자에게 노출 후 진행.
