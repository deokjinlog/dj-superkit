### `/remove-skill` 사용 예시

```bash
# 기본 (safe-rename)
/remove-skill auto-format-md
→ ~/.claude/skills/auto-format-md/ → ~/.claude/skills/auto-format-md.removed-20260525231500/

# 미리보기
/remove-skill --dry-run auto-format-md
→ rename 예정 경로 안내 + 변경 X

# Hard-delete (회복 불가)
/remove-skill auto-format-md --force
→ rm -rf ~/.claude/skills/auto-format-md/
```

### `/remove-skill` 보고 양식

**Safe-rename 성공**:
```
✅ ~/.claude/skills/<slug>/ 가 안전 정리되었습니다.

rename: ~/.claude/skills/<slug>/ → ~/.claude/skills/<slug>.removed-<timestamp>/

회복하려면: mv ~/.claude/skills/<slug>.removed-<timestamp> ~/.claude/skills/<slug>
완전 삭제하려면: rm -rf ~/.claude/skills/<slug>.removed-<timestamp>/

자동 발동이 즉시 차단됐는지 확인하려면 /reload-plugins 한 번 호출해주세요.
```

**`--force` hard-delete 성공**:
```
✅ ~/.claude/skills/<slug>/ 가 영구 삭제되었습니다.

⚠️ 회복 불가. 다시 만들려면 /new-skill <slug> "..." 로 재생성.
```

**Dry-run**:
```
ℹ️ /remove-skill <slug> 미리보기 (--dry-run, 변경 X)

기존 경로: ~/.claude/skills/<slug>/ (<크기> KB, 마지막 수정 <timestamp>)
변경 예정: ~/.claude/skills/<slug>.removed-<timestamp>/ (rename)

--force 옵션 시 hard-delete (회복 불가).

옵션 빼고 재호출하면 실제 변경 적용.
```

**Skill 부재**:
```
⚠️ ~/.claude/skills/<slug>/ 가 존재하지 않습니다.

현재 글로벌 skill 디렉토리 목록:
- <slug-1>
- <slug-2>
- ...

slug 명을 다시 확인해주세요.
```

## §5. 결정 + 대안 비교

### D-T1. `/remove-skill` default — safe-rename vs hard-delete vs interactive confirm

| 대안 | 장점 | 단점 |
|---|---|---|
| A (선택) — safe-rename (`.removed-<timestamp>`) | 회복 가능, 사용자 실수 catch | rename 후 자동 발동 차단 보장 명시 X (R-1) |
| B — hard-delete default (`rm -rf`) | 즉시 정리 | 사용자 실수 시 데이터 손실 (회수 불가) |
| C — 매 호출마다 사용자 확인 prompt | 안전성 최대 | 일관성 X — `/new-skill` 은 비인터랙티브, UX 비대칭 |

**채택 = A**. v2.6.0 의 `/new-skill --force` 백업 룰 (D-T5) 답습 + 안전 default 원칙.

### D-T2. `--force` hard-delete — Bash rm -rf 호출 방법

| 대안 | 장점 | 단점 |
|---|---|---|
| A (선택) — Bash 도구로 `rm -rf` 호출 | 1 step, 명확 | 빌더가 bash 호출 — instruction-only 패턴 약간 위반 |
| B — Write 도구로 빈 파일 덮어쓰기 | bash 없음 | rm -rf 실제 효과 X (디렉토리 안 다른 파일 남음) |

**채택 = A**. `/new-skill` 본문이 instruction-only 였지만 hard-delete 는 bash 필요. 빌더 본문 명시 "이 한 step 만 bash 사용" 으로 patch.

### D-T3. `--dry-run` preview 형식

| 대안 | 장점 | 단점 |
|---|---|---|
| A (선택) — 메인 응답 prose (변경 예정 경로 + 크기 + timestamp 안내) | 사용자 시야 즉시, 추가 파일 X | 크기/timestamp catch 위해 LS 도구 호출 |
| B — 임시 파일 (`/tmp/remove-skill-preview.txt`) | 사용자 응답 짧음 | 추가 파일 / cleanup 부담 |

**채택 = A**. `/new-skill --dry-run` 답습 패턴.

### D-T4. `/new-skill` description 좁히기 — 표현 방식

| 대안 | 장점 | 단점 |
|---|---|---|
| A (선택) — "사용자가 `/new-skill` 슬래시를 명시 호출했을 때만 발동. ..." | 자동 발동 catch 룰 약화, 명시 트리거 명확 | description 길어짐 (max 120자 룰과 충돌 가능 — 현재 약 100자) |
| B — frontmatter 에 `user-invocable: false` 필드 추가 | skill 디렉토리에서 자동 발동 차단 | commands frontmatter 가 그 필드 지원하는지 확인 안 됨 |
| C — description 통째로 짧게 ("Manual /new-skill invocation only") | 자동 발동 0 보장 | 사용자가 description 만 보고 빌더 동작 catch 어려움 |

**채택 = A**. "명시 호출 only" 톤 우선 + 자동 발동 catch 약화. 100% 보장 X — v2.6.x 후속 dogfood 검증 영역.

### D-T5. step 폭주 alert 정정 — 메시지

| 대안 | 장점 | 단점 |
|---|---|---|
| A (선택) — "그대로 진행하려면 같은 명령을 다시 호출해주세요. 분리하려면 자유 텍스트를 두 개로 나눠 두 번 호출해주세요." | `--force` 명시 제거, 사용자 옵션 명확 | 한국어 두 문장 |
| B — 단순 안내 ("step 이 많습니다. 계속 진행합니다.") | 짧음 | 사용자가 선택지 모름 |
| C — abort (강제 차단) | 회귀 0 | 사용자 마찰 ↑ — D-T2 의 warn 룰 (강제 X) 위반 |

**채택 = A**. v2.6.0 의 warn 수준 룰 답습.

### D-T6. `--force` 백업 instruction 명시화 — 3 step

| 대안 | 장점 | 단점 |
|---|---|---|
| A (선택) — Read → Write bak → Write 덮어쓰기 (3 step 명시) | Claude 도구 흐름 정확, atomic 보장 가능 | 3 step instruction 길어짐 |
| B — Bash `cp` 호출 후 Write | 1 bash + 1 Write | instruction-only 패턴 위반 (D-T1 위반) |
| C — 그대로 "Write 도구로 copy" 모호 유지 | 변경 X | 문제 그대로 (실제 copy 안 됨) |

**채택 = A**. instruction-only 답습 + atomic 보장.

### D-T7. CLAUDE.md 결합 메모 패턴

| 대안 | 장점 | 단점 |
|---|---|---|
| A (선택) — v2.6.0+ 섹션에 v2.6.1+ 분기 추가 (worktree-merge-back v2.0.4+ / v2.5.1+ 분기 패턴 답습) + `/remove-skill` 신규 섹션 | 결합 룰 history 보존 | CLAUDE.md 길어짐 |
| B — v2.6.0+ 섹션 자체 v2.6.1 룰로 덮어쓰기 | 짧음 | history 손실 |
| C — 신규 `/remove-skill` 섹션만 추가 (patch 회수 메모 X) | 짧음 | v2.6.0 catch 룰 누락 |

**채택 = A**. 다른 v2.x 결합 메모 패턴 답습.
