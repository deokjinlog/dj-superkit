## §1. 개요

`/new-skill` 슬래시 명령을 dj-superkit 의 `commands/new-skill.md` 에 신규 추가. 자유 텍스트 한 줄을 입력받아 LLM 이 자체 분해해서 글로벌 `~/.claude/skills/<slug>/SKILL.md` 1 파일을 자동 생성. expand-plan.md 패턴 답습 — instruction-only, bash 호출 없음, Read / Edit / Write 도구만 사용.

**핵심 흐름**:

```
사용자 입력 (/new-skill [<slug>] [--force] [--dry-run] "<자유 텍스트>")
   ↓
LLM 분해 (D9 의 5 단계)
   ├─ 트리거 조건 추출 → description (max 120자, 1줄)
   ├─ 수행 동작 추출 → 본문 instruction (bite-sized step 1~7)
   ├─ 슬러그 자동 생성 (또는 명시 인자 사용)
   └─ user-invocable 결정 (기본 true)
   ↓
사전 검증
   ├─ 기존 파일 충돌 (--force 없으면 abort)
   ├─ description 휴리스틱 (D7)
   └─ 비밀값 / 토큰 / 하드코딩 경로 거부 (C6)
   ↓
Write (또는 dry-run 시 preview 출력)
   ↓
사용자 보고 (D10 양식)
```

## §3. 데이터 흐름

### 입력 파싱

`$ARGUMENTS` 토큰 분해 (단순 split + 휴리스틱):

```
/new-skill auto-format-md --dry-run "사용자가 .md 파일 저장하면 ..."
↓
- slug 인자: "auto-format-md" (kebab-case 패턴 매치)
- 플래그: ["--dry-run"]
- 자유 텍스트: "사용자가 .md 파일 저장하면 ..."
```

룰 — 첫 토큰이 kebab-case (소문자 + 하이픈) 매치 시 slug 인자, 그 외는 자유 텍스트. `--force` / `--dry-run` 은 어디 있어도 플래그로 인식.

### LLM 분해 (5 단계)

1. **트리거 조건** = "사용자가 ~ 했을 때" / "~ 하면" 패턴 catch → "Use when ..." 형식으로 변환 (1줄 max 120자).
2. **수행 동작** = 동사구 catch → bite-sized step 1~7 (`### Step N — <헤더>` + 짧은 설명 1~3줄).
3. **슬러그** = 명시 인자 우선. 없으면 자유 텍스트의 핵심 명사구 catch + 영문 kebab-case 변환 (메인 자체 추론).
4. **user-invocable** = "메뉴에 띄우지 마" / "숨겨" / "감춰" 단서 catch 시 false, 외 true.
5. **충돌 검증** = `~/.claude/skills/<slug>/` 디렉토리 존재 여부 LS 도구 확인.

### Write / Dry-run 분기

```
--dry-run 명시?
   ├─ Y → 메인 응답으로 preview 출력 (frontmatter + 본문). Write 도구 호출 X.
   └─ N → Write 도구 호출
       ├─ 기존 파일 존재 + --force X → abort + 기존 파일 경로 안내
       ├─ 기존 파일 존재 + --force Y → 백업 (`<slug>/SKILL.md.bak-<timestamp>`) + 덮어쓰기
       └─ 기존 X → mkdir -p `~/.claude/skills/<slug>/` + Write SKILL.md
```

## §4. 외부 인터페이스

### 슬래시 명령 frontmatter

```yaml
---
description: "자유 텍스트 한 줄을 받아 글로벌 ~/.claude/skills/<slug>/SKILL.md 1 파일로 자동 생성하는 skill 빌더. frontmatter + 본문 instruction 까지 LLM 이 분해해서 한 턴에 생성."
argument-hint: "[<slug>] [--force] [--dry-run] <자유 텍스트 설명>"
---
```

### 사용 예시

```bash
# 기본 (slug 자동 생성)
/new-skill "사용자가 .md 파일 저장하면 prettier 로 자동 포맷하는 skill"

# slug 명시
/new-skill auto-format-md "사용자가 ..."

# 미리보기 (Write X)
/new-skill --dry-run "사용자가 ..."

# 덮어쓰기 (백업 후)
/new-skill auto-format-md --force "사용자가 ..."
```

### 완료 후 보고 양식 (D10 답습)

성공:
```
✅ /<slug> skill 이 ~/.claude/skills/<slug>/SKILL.md 에 생성되었습니다.

발동 조건: <description 1줄 요약>

저장 즉시 사용 가능합니다 (skill hot-reload).
- 사용자 명시 호출: /<slug>
- 자동 발동: Claude 가 description 매칭 판단 시
```

Dry-run:
```
ℹ️ /<slug> skill 미리보기 (--dry-run, Write X)

[frontmatter + 본문 instruction preview]

옵션 빼고 재호출하면 실제 Write 합니다.
```

충돌 + `--force` X:
```
⚠️ ~/.claude/skills/<slug>/SKILL.md 가 이미 존재합니다.

기존 파일: <경로> (<크기> KB)
덮어쓰려면 `--force` 옵션 추가 (백업 후 덮어쓰기).
다른 slug 로 만들려면 첫 인자에 명시 (예: /new-skill new-slug "...").
```

## §5. 결정 + 대안 비교

### D-T1. instruction-only vs bash 호출 혼합

| 대안 | 장점 | 단점 |
|---|---|---|
| A (선택) — instruction-only (expand-plan.md 답습) | 메인 latency 낮음, 사용자 시야 투명, hooks 의존 X | 슬러그 자동생성 / 충돌 검증 모두 LLM 자체 판단 → 정확도 변동 |
| B — bash 호출 혼합 (slug 추출 helper / LS 충돌 검증) | 슬러그 / 충돌 검증 결정론적 | helper 파일 추가 / 의존성 ↑ / 빌더 자체 설치 시점 마찰 |

**채택 = A**. requirements C5 ("메인 컨텍스트 폭주 X") 와 정합. js-ralph expand-plan.md 패턴 답습.

### D-T2. LLM 분해 step 수 권장 범위

| 대안 | 장점 | 단점 |
|---|---|---|
| A (선택) — 1~7 step | bite-size 룰 답습, contributor 가 회상 가능 | 자유 텍스트가 복잡하면 어느 step 묶음 / 분리 판단 LLM 변동 |
| B — 무제한 | 큰 skill 도 자유 분해 | step 폭주 위험 (10+ step → 메인이 회상 못 함) |

**채택 = A**. dj-superkit 의 다른 skill 본문도 대부분 1~7 step.

### D-T3. description 길이 자동 검증 (D7)

| 대안 | 장점 | 단점 |
|---|---|---|
| A (선택) — 120자 + 1줄 max, 위반 시 자동 trim + 알림 | 자동 발동 게이트 정확도 보장, 사용자 마찰 X | trim 결과가 의도 미스 시 사용자 catch 필요 |
| B — soft warn (강제 X) | 사용자 자유도 ↑ | 오발동 회귀 위험 (긴 description → 발동 판단 비용 ↑) |
| C — hard abort (max 위반 시 거부) | 회귀 0 | 사용자 마찰 ↑ (재시도 강제) |

**채택 = A**. 자동 trim 후 사용자에게 한 줄 알림이 균형.

### D-T4. 슬러그 충돌 검증 범위

| 대안 | 장점 | 단점 |
|---|---|---|
| A (선택) — 글로벌 `~/.claude/skills/` 만 | 빠름, 빌더가 Write 할 위치만 검사 | 사용자 환경의 다른 플러그인 / 프로젝트 skill 과 이름 겹칠 위험 (자동 발동 우선순위 변동) |
| B — 모든 skill 디렉토리 (plugin cache + 프로젝트 + 글로벌) | 회귀 0 | LS 도구 3+ 호출, 빌더 latency ↑ |

**채택 = A**. 빌더 Write 대상이 글로벌이므로 직접 충돌만 검증. 자동 발동 우선순위는 사용자 catch 영역.

### D-T5. `--force` 시 백업 vs 직접 덮어쓰기

| 대안 | 장점 | 단점 |
|---|---|---|
| A (선택) — 백업 (`SKILL.md.bak-<timestamp>`) 후 덮어쓰기 | 사용자 회수 가능, 회복력 ↑ | 백업 파일 누적 (cleanup 사용자 부담) |
| B — 직접 덮어쓰기 (백업 X) | 군더더기 X | 사용자 실수 시 복구 X |

**채택 = A**. requirements C1 (안전성) 정합. cleanup 은 사용자 명시 의도 영역.

### D-T6. dry-run preview 출력 위치

| 대안 | 장점 | 단점 |
|---|---|---|
| A (선택) — 메인 응답 prose | 사용자 시야 즉시, 추가 파일 X | preview 길면 사용자 응답 길이 ↑ |
| B — 임시 파일 (`/tmp/new-skill-preview-<timestamp>.md`) | 사용자 응답 짧음 | 추가 파일 / cleanup 부담 |

**채택 = A**. 빌더는 한 턴에 끝나는 게 핵심 UX.

### D-T7. 빌더 본문 instruction 의 사용자 안내 톤

| 대안 | 장점 | 단점 |
|---|---|---|
| A (선택) — 한국어 친화 (v2.4+ A-1~A-5 룰 답습) | dj-superkit 톤 일관성 | 영어 contributor 가 본문 못 읽음 |
| B — 영어 본문 | 글로벌 호환 | dj-superkit 톤 불일치 |

**채택 = A**. dj-superkit 의 모든 신규 slash command 가 한국어 톤. v2.4+ CLAUDE.md 룰 강제.

## §7. 테스트 전략

instruction-only 빌더라 단위 테스트 어려움. **dogfood 시나리오** 로 검증:

| 시나리오 | 입력 | 기대 |
|---|---|---|
| T-1 happy path (slug 자동) | `/new-skill "사용자가 X 하면 Y 와 Z 차례로 수행"` | `~/.claude/skills/x-handler/SKILL.md` (또는 유사 slug) 생성 + D10 보고 |
| T-2 slug 명시 | `/new-skill my-helper "..."` | `~/.claude/skills/my-helper/SKILL.md` 생성 |
| T-3 dry-run | `/new-skill --dry-run "..."` | 메인 응답 preview + Write X |
| T-4 충돌 + force X | (T-1 직후 재호출) | abort + 기존 파일 안내 |
| T-5 충돌 + --force | `/new-skill x-handler --force "..."` | 백업 (`.bak-<timestamp>`) + 덮어쓰기 |
| T-6 비밀값 catch | `/new-skill "AWS_KEY=AKIA... 를 박아"` | abort + 비밀값 catch 안내 |
| T-7 description 길이 위반 | `/new-skill "150자 넘는 길고 모호한 설명..."` | 자동 trim + 사용자 알림 |
| T-8 hot-reload 검증 | T-1 결과 직후 `/<slug>` 호출 | skill 본문 로드 + 동작 |

T-1 / T-3 / T-4 / T-8 = critical 시나리오. T-5 / T-6 / T-7 = edge case (대표성).
