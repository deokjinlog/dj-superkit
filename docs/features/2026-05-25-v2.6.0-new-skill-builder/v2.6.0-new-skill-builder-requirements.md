## §1. 배경

js-super 의 skill 자산은 v2.5.2 기준 35개. 새 skill 을 추가하려면 contributor 가 직접 `skills/<slug>/SKILL.md` 의 frontmatter (name / description / user-invocable) + 본문 instruction 을 손으로 작성해야 함. 이 작업은 다음 마찰을 만듦.

- frontmatter 의 `description` 필드는 자동 발동 트리거 게이트 역할 — 1줄로 명확하게 써야 하지만 contributor 가 자주 길게 / 모호하게 작성 → 오발동 또는 미발동 회귀.
- 본문 instruction 의 bite-sized step 분해도 contributor 자체 판단 — 노련하지 않으면 step 수 폭주 또는 한 step 에 다중 책임.
- 빌트인 인터랙티브 UI 부재 (`/agents` 는 sub-agent 전용. 출처: code.claude.com/docs/en/agent-sdk/slash-commands).
- skill hot-reload 는 지원되지만 (command 와 달리 즉시 로드) 사용자가 빌더 한 줄로 만들 방법이 없어 매번 수동 작성.

→ **`/new-skill` 슬래시 빌더** 가 자유 텍스트 한 줄을 받아 frontmatter + 본문 instruction 으로 자동 분해 + Write 하면, skill 자산 누적 마찰이 사라짐. js-ralph 의 `/expand-plan` 과 동일한 "알잘딸깍센" 컨셉.

## §2. 핵심 결정

### D1. 빌더 위치: js-super 안 (`commands/new-skill.md`)

- v2.6.0 minor 릴리즈로 동봉 → 다음 사용자 자동 받음
- 사용자 명시 선택 (Q1 답변 "1")
- 별도 플러그인 (META-BUILDER §6 원문 의도) 또는 글로벌 (`~/.claude/commands/`) 대안은 v2.6.x 후속에서 재검토 여지

### D2. 생성된 skill 저장 위치: 글로벌 (`~/.claude/skills/<slug>/SKILL.md`)

- Claude Code 가 user-scope skill 로 자동 인식 + hot-reload 지원 (모든 프로젝트에서 호출 가능)
- 사용자 명시 선택 (Q1 답변 "C")
- 사용자가 직접 작성한 skill 자산이 사용자 환경에 누적 + cwd 무관 활용 가능
- 플러그인 안 (`<plugin>/skills/`) 은 cache 안이라 plugin update 시 휘발 위험 — 추천 X

### D3. 슬래시 이름: `/new-skill`

- META-BUILDER §6 default 답습. `/create-skill` / `/scaffold-skill` 대안 채택 X
- 짧고 동사 + 명사 패턴

### D4. 슬러그 자동생성 규칙

- 사용자가 첫 인자로 명시 시 그대로 사용 (예: `/new-skill auto-format-md "이런 skill 만들어"`)
- 명시 X 인 경우 자유 텍스트에서 핵심 명사구 추출 + 영문 kebab-case 변환
- 모호 시 prose 한 줄 묻고 진행 (메인 자체 판단)

### D5. `--force` 옵션 (덮어쓰기 명시 opt-in)

- 기본 동작: 동일 이름 skill 존재 시 abort + 기존 파일 경로 안내
- `--force` 명시 시: 백업 (`<slug>/SKILL.md.bak-<timestamp>`) + 덮어쓰기
- 안전 default — 사용자 의도 없는 덮어쓰기 차단

### D6. `--dry-run` 옵션 (Write 전 미리보기)

- 명시 시 실제 Write X — frontmatter + 본문 preview 만 메인 응답으로 출력
- 사용자 검토 후 옵션 빼고 재호출하면 실제 Write
- skill 작성 후 회수 어려움 → preview 단계로 마찰 ↓

### D7. description 자동 검증 휴리스틱

빌더가 Write 직전 다음 3 휴리스틱으로 description 검증:

| 휴리스틱 | 룰 | 위반 시 |
|---|---|---|
| 길이 | max 120자, 1줄 (개행 X) | 자동 trim + 사용자에게 한 줄 알림 |
| 트리거 명확성 | "Use when ..." / "When the user ..." 패턴 권장 | prose 안내 + 그대로 진행 (강제 X) |
| 동사 + 객체 | 능동태 동사로 시작 | prose 안내 + 그대로 진행 (강제 X) |

오발동 / 미발동 회귀 방지 목적. 강제 차단은 사용자 마찰만 키우므로 알림 수준.

### D8. 빌더 자체는 command 로 (skill 로 만들기 X)

- META-BUILDER §5 명시 — 메타 도구는 명시 호출만 받아야 안전
- skill 로 만들면 자동 발동 사고 위험 ("사용자가 평소 대화에서 skill 만들어달라고 안 했는데도 발동")
- expand-plan.md 패턴 (instruction-only, bash 없음, frontmatter + 본문 instruction) 답습

### D9. 자동 분해 로직 (5 단계)

빌더가 자유 텍스트 받으면:

1. **트리거 조건 추출** → description (1줄)
2. **수행 동작 추출** → 본문 instruction (bite-sized step 1~7)
3. **슬러그 자동 생성** (D4 룰)
4. **`user-invocable` 결정** — 입력에 "메뉴에 띄우지 마" / "숨겨" 같은 단서 있으면 false, 기본 true
5. **`~/.claude/skills/<slug>/SKILL.md` Write** (D5 / D6 분기 검증 후)

### D10. 완료 후 사용자 보고 양식

```
✅ /<slug> skill 이 ~/.claude/skills/<slug>/SKILL.md 에 생성되었습니다.

발동 조건: <description 1줄 요약>

저장 즉시 사용 가능합니다 (skill hot-reload).
- 사용자 명시 호출: /<slug>
- 자동 발동: Claude 가 description 매칭 판단 시
```

## §3. 우려 / 해결

### C1. 빌더가 사용자 시스템 글로벌에 파일 박는 게 안전한가?

- **해결**: D5 `--force` 안 주면 동일 이름 abort 보장. 글로벌 `~/.claude/skills/` 는 사용자 본인이 명시 활용하는 영역이라 의도 부합. D6 `--dry-run` 으로 사전 검토 가능.

### C2. description 자동 검증이 contributor 자유도 제약?

- **해결**: D7 의 휴리스틱 3건 중 길이만 자동 trim, 나머지 2건은 알림 수준 (강제 차단 X). 사용자가 명시 의도면 그대로 통과.

### C3. 자유 텍스트 한 줄에서 트리거 / 동작 분리가 어려운 입력?

- **해결**: 메인이 한 줄 묻고 진행 (예: "트리거 조건과 수행 동작을 분리해 알려주세요"). 무한 clarifying 방지 — 1회만.

### C4. 글로벌 저장 시 다른 PC / 동료와 공유 안 됨?

- **해결**: 명시 의도 — 사용자별 자산 누적. 공유 필요 시 사용자가 별도 플러그인 / 저장소로 옮기는 건 별도 흐름. v2.6.x 후속에서 "skill export" 명령 후보.

### C5. 빌더가 메인 컨텍스트 폭주 시켜 다른 작업 차단?

- **해결**: instruction-only, bash 없음 (expand-plan.md 패턴 답습). Read / Edit / Write 도구만 사용. 메인 latency 거의 0.

### C6. skill 본문에 사용자 자유 텍스트 그대로 박혀 안티 패턴 (비밀값 / 토큰) 박힘?

- **해결**: 빌더 본문에 명시 거부 룰 — 비밀값 / 토큰 / 하드코딩 경로 catch 시 abort + 사용자 안내.
