## 배경

v2.4.0 에서 한국어 친화 톤 룰 (A-1~A-5) 을 CLAUDE.md 에 박고, 4 PRD + auto-* 8 skill 의 ℹ️ 안내 라인 18 줄을 직접 정리했습니다. 그런데도 사용자가 본 v2.4.2 진입 직전 두 가지 catch 를 줬습니다.

첫째, 단어 자체가 어렵다. 사용자 인용:

> "그 단어들이 너무 자기들만 아는 단어를 써댔어"

`dispatch`, `boilerplate`, `manifest`, `slug` 같은 영어 단어를 한국어 안에 그대로 섞어 쓰는 패턴이 v2.4 매핑 8 항목으로도 못 잡힌 영역까지 잔존.

둘째, 톤보다 설명 방식 자체가 어렵다. 사용자 인용:

> "한국어 친화 톤이라긴보다 그냥 설명자체가 너무 개판이야 너무 고지식하게 하니까 이해하기 힘들어"

표 다발 / `## ~ 상황` 같은 보고서 형식 헤더 / `옵션 A / 옵션 B` 메뉴식 묻기 / 글머리표 다발 — 이게 다 너무 형식적이라 어려움. 친구한테 말하듯 자연스러운 줄글이 필요.

셋째, 핵심 통찰. 사용자 인용:

> "근데 이거 플러그인이라 우리꺼 CLAUDE.md 건드려봤자 저쪽에 못쓰는거 아닌가?"

저장소 루트의 CLAUDE.md 는 contributor 가이드라서 플러그인 cache 에 포함은 되지만 자동 로드는 안 됨. 사용자가 자기 프로젝트에서 dj-superkit 플러그인 쓸 때 우리 CLAUDE.md 는 메인 시야에 안 들어옵니다. 즉 v2.4 의 A 항목 룰을 CLAUDE.md 에만 박아둔 게 효과 미흡했던 진짜 이유 — 사용자 환경에 안 가서.

세 catch 가 합치된 결론은 명확합니다. CLAUDE.md 룰 추가 / 강화로 풀 게 아니라, 사용자 환경에 실제로 전달되는 영역 (skill body + commands/*.md) 의 실제 본문 자체를 자연스러운 줄글로 광범위하게 풀어써야 합니다.

## 핵심 결정

### D1: CLAUDE.md 안 건드림 (예외 1건)

v2.4 의 A 항목 (CLAUDE.md A-1~A-5 + 단어 매핑) 은 그대로 유지합니다. 우리 자체 개발 시 (이 세션 같은) 메인 시야의 자기 참조용으론 여전히 쓸모 있고, 사용자 환경엔 어차피 안 전달되니 손해도 없습니다.

톤 룰 추가 / 강화는 본 v2.4.2 에서 박지 않습니다. 대신 사용자 환경에 진짜 전달되는 영역만 손봅니다.

**예외 1건 — contributor 안전망 메타 노트**: 본 v2.4.2 에서 CLAUDE.md 최상단 (dj-superkit 메모 시작점) 에 "## ⚠️ 본 CLAUDE.md 의 적용 범위" 섹션을 추가합니다. 미래 세션이 또 같은 실수 (= 사용자 영향 룰을 CLAUDE.md 에 박기) 를 반복하지 않도록 catch 안내. 톤 룰 추가가 아니라 contributor 안전망이라 D1 정신과 합치합니다. 사용자 catch 인용:

> "이거 똑같은 실수 안하게 우리 CLAUDE.md에 이부분 잘 적어줄래? 사용자가 볼 CLAUDE.md가 아니니까 유의하라고 전체 공통 로직이 아니라고.."

콜론 다발 (`결과: X / 다음: Y`) 보다 완전 문장 풀어쓰기.

## 우려와 해결

### 우려 1 — 표 / 형식 헤더 / 글머리표 자체가 skill body 에선 LLM prompt 가공용으로 필요한 영역

skill body 는 메인 LLM 이 prompt 로 받는 가공 자료라 표 / 글머리표가 효과적인 정보 전달 형식. 사용자 노출 영역만 손대고 LLM prompt 영역 (Process / Anti-Patterns 표의 영어 식별자 / 코드 패턴) 은 그대로 둡니다.

기준: 메인이 사용자에게 turn 중에 출력하는 영역만. 즉 ℹ️ / Step prose / Why-How / 사용자 안내 영역. 영어 식별자 / 함수명 / 파일 경로 / Anti-Patterns 표의 코드 셀은 그대로.

### 우려 2 — commands/*.md 의 일관성

v2.4 의 sync-html.md 에 `--check` 옵션만 추가했고 나머지 본문은 v2.2.2 시점 그대로. 본 v2.4.2 에서 전체 정리하면 일관성 회복. 일부만 정리하면 파일마다 톤이 들쭉날쭉.

해결: 15 파일 전부 정리. 작업량 늘어나지만 일관성이 핵심.

### 우려 3 — 사용자 노출 영역 식별 기준이 모호할 수 있음

skill body 의 어떤 라인이 사용자 노출인지 모호한 경우 있음. 예: Anti-Patterns 표 안의 한글 셀 — 사용자에게 직접 노출 안 되지만 메인이 보고 사용자에게 풀어 옮길 가능성.

해결: 안전하게 풀어쓰는 쪽으로. 형식 변경은 최소, 단어 매핑은 적극.
