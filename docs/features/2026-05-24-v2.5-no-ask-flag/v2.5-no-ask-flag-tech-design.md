### 1.2 분기 boilerplate 본문 schema

각 skill body 의 "사용자 질문 룰" 섹션 직후에 `--no-ask` 분기 sub-section 추가. 표준 본문:

```markdown
### 예외 — `--no-ask` 플래그 (v2.5+)

사용자가 슬래시 명령에 `--no-ask` 토큰을 **명시** 한 경우에만 진입.
메인 자체 판단으로 활성화 X.

- 모든 사용자 질문을 prose (메인 turn 자유 텍스트) 로 처리
- `AskUserQuestion` 도구 호출 **0 보장**
- 게이트 자체는 살아 있음 — 사용자 prose 응답 기다림
- 알람 fire X (사용자가 명시 invoke 했으니 인지 가정)

#### skill 진입 시 1회 boilerplate

skill 진입 직후 다음 한 줄을 prose 로 출력:

> ℹ️ `--no-ask` 모드 진입 — AskUserQuestion 도구 호출 X, 응답 알람 X. 백그라운드 작업 중이면 응답 시점을 직접 체크해주세요.

#### 위험 명령 진입 직전 보강

critical 7 케이스 (파일 삭제 / `git push --force` / DB migration / mass commit / 외부 메시지 등) 실행 직전에는 다음 한 줄을 prose 로 출력:

> ⚠️ 위험 명령 진입 — 응답 기다림. 백그라운드 작업 중이면 직접 catch 해주세요.

`⚠️` 마커 + 별도 줄로 일반 prose 보다 두드러지게.

#### auto-\* 4 skill 추가 룰

내부 escalation 경로 (BLOCKED 자가복구 실패 / critical 7 재질문 / Other 모호 응답 재질문) 에서도 `AskUserQuestion` 호출 **0 보장**. clarifying Q 자체는 그대로 prose 유지.
```

이 표준 본문이 8 skill 의 SKILL.md 에 동일 박힘.

### 1.3 commands/\*.md 본문 보강 schema

각 슬래시 명령 본문 끝에 `--no-ask` 플래그 섹션 추가:

```markdown
## `--no-ask` 플래그 (v2.5+)

`AskUserQuestion` 도구가 느리거나 불안정할 때, 도구 호출을 완전히 우회하고 싶을 때 사용:

`/brainstorm <slug> --no-ask`

질문은 그대로 받지만 메인 에이전트가 채팅 메시지 (prose) 형식으로 묻습니다. 사용자도 채팅으로 응답하면 됩니다. 알람 fire X — 백그라운드 작업 중이면 응답 시점을 직접 체크해야 합니다.

플래그 위치 자유 (`<slug> --no-ask` 또는 `--no-ask <slug>` 모두 가능).
```

각 슬래시 명령 별 첫 라인 (`/brainstorm`) 만 다름. 나머지 본문은 동일.

## 핵심 결정 + 대안 비교

### D-1 (§1.2) — 분기 본문 sub-section 위치

대안:
- A. "사용자 질문 룰" 직후 sub-section
- B. skill body 끝에 별도 섹션
- C. 별도 파일로 분리 (예: `--no-ask-rules.md`) + skill body 에서 reference

선택: **A**. 사용자 질문 룰과 의미 결합이 강함. 메인이 사용자 질문 결정 시 한 곳에서 룰 + 분기 모두 catch. B 는 룰 ↔ 분기 거리 멀어 catch 손실. C 는 self-contained 손상.

### D-2 (§1.2) — auto-\* 추가 룰 박는 위치

대안:
- A. PRD 4 와 동일 분기 본문 + auto-\* 만 추가 sub-section
- B. auto-\* 4 만 별도 분기 본문 (PRD 와 schema 다름)

선택: **A**. 분기 본문 schema 통일 → 회귀 catch grep 단순화. auto-\* 추가 룰 (escalation 도구 호출 0) 만 sub-section 으로.

### D-3 (§1.2) — 위험 명령 안내 마커

대안:
- A. `⚠️` 마커 + 별도 줄
- B. `🚨` 마커 + 별도 줄
- C. `⛔` 마커 + 별도 줄
- D. 마커 없이 굵게

선택: **A**. `⚠️` 가 v2.4 의 다른 위험 안내 (`/sync-html --rebuild` 경고 등) 와 일관성. v2.4 톤 룰의 "✅ / → / ⚠️ 한 줄 안에 허용" 답습.

### D-4 (§2.1) — 변경 영역 vs Wave 분할

대안:
- A. Wave 0 (CLAUDE.md 결합 메모) → Wave 1 (PRD 4 skill) → Wave 2 (auto-\* 4 skill) → Wave 3 (commands 8) → Wave 4 (6 manifest)
- B. atomic 1 commit (17 본문 + 6 manifest)
- C. PRD 4 + commands 4 / auto-\* 4 + commands 4 / CLAUDE.md + manifest — 3 wave

선택: **A**. v2.4.2 와 동일 패턴 (5 wave + Wave 0). 사용자 catch 익숙. wave 별 회귀 catch 가능. B 는 atomic 너무 큼 → review 부담. C 는 wave 간 회귀 catch 단위 어색.

### D-5 (§5.2) — 회귀 catch grep level

대안:
- A. 메인 dogfood (release 직전 메인이 grep 실행)
- B. CI workflow 추가 (GitHub Actions)
- C. Git pre-commit hook

선택: **A**. dj-superkit 가 plugin 이라 CI / hook 인프라 없음 (사용자 환경 영향 0 보장). 메인 dogfood 가 자연스러움. B / C 는 plugin 사용자 환경에 외부 의존 추가됨 — zero-dependency 룰 위반.
