v2.5 — --no-ask 플래그 (AskUserQuestion 도구 우회)

Mode: socratic-free-form

Contents

배경

v2.0.3+ 에서 8 skill body (PRD 4 + auto-* 4) 에 박힌 boilerplate ("사용자 질문 룰" + v2.3.5+ "AskUserQuestion 도구 우선") 가 모든 사용자 질문을 AskUserQuestion 도구 호출로 강제. 의도는 알람 누락 방지 (Notification.elicitation_dialog 매처 → repeat-alert.sh 발화).

하지만 사용자 신고:

"요즘 AskUseQuestion 이게 꽤 불안정해 막 대답이 엄청 느려.. 그래서 그.. 예를들어 /brainstorming --no-ask 하면 그 AskUseQuestion 도구를 절대 하지않는 버전을 만들고 싶어"

AskUserQuestion 도구 응답이 느리거나 가끔 안 돌아오는 케이스 신고. 4 단계 게이트 + 흐름 중간 clarifying Q 누적되면 사용자 마찰 ↑.

본 release 는 옵트인 플래그로 우회 — 도구 호출 자체를 0 으로 만들고, 질문은 prose (메인 turn 채팅 메시지) 로 처리.

핵심 결정

D1

D1 — 적용 범위 (8 skill)

영역--no-ask 적용
brainstorming / designing-direction / writing-plans / executing-plans O
auto-brainstorming / auto-designing-direction / auto-writing-plans / auto-executing-plans O
fast-tasks X— Socratic 2~3 Q × N 형식이 본질 기능, --no-ask 와 충돌
worktree-merge-back X— destructive 작업 후처리 게이트가 본질. default yes 안 됨
og-* (upstream mirror) X— 보존 룰
D2

D2 — 플래그 이름 --no-ask

사용자 신고 표현 답습 (/brainstorming --no-ask). 짧고 의미 명확. 영어 식별자 보존 룰 따름.

대안 (--prose-only, --quick, --silent) 보다 사용자 인지 비용 최저.

D3

D3 — 플래그 위치 자유

shell
/brainstorm <slug> --no-ask
/brainstorm --no-ask <slug>

slug 앞/뒤 모두 허용. argument parser 강건성 위해 메인이 --no-ask 토큰을 별도 검출.

D4

D4 — critical 케이스 처리 = 모든 case prose + 안내 보강

--no-ask 진입 시 v2.3.5+ critical 7 case (파일 삭제 / git push --force / DB migration / mass commit / 외부 메시지 등 위험 명령) 도 prose 로 묻습니다. 도구 호출 0 일관성 우선.

단 안전성 보강: 위험 명령 진입 직전 prose 에 다음 한 줄 boilerplate 박음.

⚠ boilerplate
⚠️ 위험 명령 진입 — 응답 기다림. 백그라운드 작업 중이면 직접 catch 해주세요.

사용자가 명시적으로 --no-ask invoke 했으니 알람 X 인지 가정. 단 위험 명령은 catch 손실 영향이 크므로 prose 안내 라인을 다른 일반 질문보다 두드러지게 (⚠️ 마커 + 별도 줄).

D5

D5 — 알람 X 안내 boilerplate

skill 진입 시 1회만 박음 (turn 마다 반복 X).

ℹ boilerplate
ℹ️ --no-ask 모드 진입 — AskUserQuestion 도구 호출 X, 응답 알람 X. 백그라운드 작업 중이면 응답 시점을 직접 체크해주세요.

skill body 의 "Announce at start" 라인 직후 또는 첫 사용자 질문 prose 직전에 박음 (skill 별 자연스러운 위치 선택).

D6

D6 — 플래그 persistence = 매 invoke 명시

세션 전체 지속 X. 매 슬래시 invoke 마다 명시 (/brainstorm <slug> --no-ask). 사용자 통제권 우선.

D7

D7 — ad-hoc 메인 turn 결정 = 구현 후 dogfood

skill body 외 흐름 중간에 메인이 ad-hoc 결정 요청하는 경우 (이렇게 진행할까요? 류) --no-ask 컨텍스트 인지하고 prose 전환 가능한지는 구현 후 dogfood 단계에서 검증. 1차 release 핵심 범위 아님.

우려 / 해결

R-1 R-1 — 도구 호출 0 보장 깨짐

위험

위험: 8 skill body 분기 추가 후에도 메인이 무의식 중 AskUserQuestion 호출.

해결

해결:

  • 8 skill body 분기 명시 — "도구 호출 0 보장" 문구 반복
  • grep 회귀 catch (CLAUDE.md 결합 메모 안 박음) — --no-ask 플래그 매치 + 분기 본문 매치 둘 다 검증

R-2 R-2 — 게이트 자체 skip 회귀

위험

위험: --no-ask 진입 시 메인이 "도구도 X 게이트도 X" 로 잘못 해석 → 자동 진행.

해결

해결:

  • 분기 본문에 "게이트는 살아 있음 — 사용자 prose 응답 기다림" 명시
  • 안티 패턴 표에 "게이트 skip" 명시

R-3 R-3 — auto-* 내부 escalation 회귀

위험

위험: auto-* skill 의 내부 escalation (BLOCKED 자가복구 실패 / critical 7 재질문 / Other 모호 응답 재질문) 에서 도구 호출 잔존.

해결

해결:

  • auto-* 4 skill 분기 본문에 "내부 escalation 경로도 도구 호출 0 보장" 명시
  • clarifying Q 자체는 그대로 prose 유지 (기존 auto-* 의 prose default 답습)

R-4 R-4 — 위험 명령 사용자 catch 실패

위험

위험: 알람 X 라 백그라운드 작업 중이면 사용자가 위험 명령 진입 직전 prose 를 놓침 → 의도치 않은 실행.

해결

해결: D4 의 ⚠️ 위험 명령 진입 boilerplate. 일반 prose 보다 두드러지게.

R-5 R-5 — 플래그 parsing 약함

위험

위험: slug 위치 / 공백 / 따옴표 등 parsing edge case 실패 → 사용자가 --no-ask 명시했는데 인식 못 함.

해결

해결:

  • 메인이 slug argument 토큰화 시 --no-ask 별도 검출 (if "--no-ask" in args.split() 패턴)
  • commands/*.md 본문에 "위치 자유 — slug 앞/뒤 모두 허용" 명시

R-6 R-6 — og-* 회귀

위험

위험: upstream mirror 의 og-* 3 skill (og-brainstorming / og-designing-direction / og-writing-plans / og-executing-plans) 에 실수로 --no-ask 분기 추가.

해결

해결:

  • og-* 절대 변경 X (CLAUDE.md 보존 룰)
  • 회귀 catch grep — og-* SKILL.md 에 --no-ask 매치 0 보장

R-7 R-7 — 사용자 무인지 활성화

위험

위험: 메인이 사용자 의도 아닌 상황에서 임의로 --no-ask 분기 활성화 (예: 빠른 답변 의도로 자동 적용).

해결

해결:

  • 분기 진입 조건 = "사용자가 슬래시 명령에 --no-ask 토큰을 명시 했을 때만"
  • 메인 자체 판단으로 활성화 X — 안티 패턴 표 명시
