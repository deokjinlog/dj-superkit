## 1. 아키텍처 개요

신규 skill `worktree-merge-back` (단일 SKILL.md body) + slash command `/worktree-merge-back` (1줄 wrapper) + 3 fixture (메타 dogfood README 만). git CLI 조작만 하고 MD / DB / 외부 API 무관.

흐름 (Step 1~5):

```
사용자 invoke
   │
   ▼
[Guard] git worktree list + cwd 비교
   ├─ main worktree / non-worktree → HARD-GATE 차단 + skill 종료
   └─ feature worktree 확인 → 진행
   │
   ▼
[Step 1] working tree 변경 검사
   ├─ clean → Step 2 자동
   └─ dirty → AskUserQuestion [지금 commit / 직접 처리 후 재호출]
   │
   ▼
[Step 2] parent worktree 추론 (git worktree list)
   └─ 추론 실패 → 명시 차단 + 안내
   │
   ▼
[Step 3] git fetch origin + git merge origin/<parent>
   ├─ clean → Step 4 자동
   └─ conflict → 충돌 파일 노출 + AskUserQuestion [수동 해결 완료 / abort]
   │
   ▼
[Step 4] git -C <parent-path> merge --no-ff <feature-branch>
   └─ 충돌 0 보장 (Step 3 흡수 완료)
   │
   ▼
[Step 5] 사후 처리 게이트 (multi-question batching, 모두 default no)
   ├─ 워크트리 제거?
   ├─ 브랜치 삭제?
   └─ parent push?
```

핵심 안전성:
- Guard 가 1차 방어선 — main 워크트리에서 호출 시 차단
- Step 3 의 "merge down before merging up" 패턴이 2차 방어선 — 충돌은 항상 feature sandbox 에서만
- 모든 destructive / remote 영향 작업은 명시 AskUserQuestion default no

## 5. 핵심 결정 + 대안 비교

### D-T1. Skill body 자체 작성 vs 기존 skill 확장

| 옵션 | 장점 | 단점 |
|---|---|---|
| **신규 skill** ← 채택 | 단일 책임 / 자동 발동 X (명시 invoke 만) / 사용자 의도 명확 분기점 보존 | 신규 skill dir + slash command 1개 추가 |
| `setting-up-worktrees` 에 머지 흐름 추가 | 1 skill 안에 lifecycle 통합 | 책임 비대 / 자동 발동 위험 / "create" 와 "merge back" 의도 차원 다름 |
| `finishing-a-development-branch` 에 통합 | finishing 흐름과 자연 연결 | finishing 은 1인 개발 슬림 (테스트 게이트 + 종료 메시지) 의도 — 머지 자동화는 scope 초과 |

**근거**: 백로그 D1 채택. 사용자 결정 강조 (자동 발동 X).

### D-T2. Worktree 제약 검출 방법 — deterministic helper vs LLM-judged

| 옵션 | 장점 | 단점 |
|---|---|---|
| **deterministic** (`git worktree list` + cwd 비교) ← 채택 | 결정적 / false positive 0 / 빠름 | 비표준 worktree 구조 시 추론 실패 (→ 명시 차단으로 처리) |
| LLM-judged (메인이 컨텍스트로 판단) | 유연 (nested worktree 등 cover 가능) | false negative 위험 (main 더럽힘 = catastrophic) |
| frontmatter 메타 (`.js-super/worktree-meta.json`) | 명시적 / 추론 불필요 | 사용자가 매번 파일 생성 부담 / setting-up-worktrees 와 결합 비용 |

**근거**: 본 skill 의 안전성이 worktree 제약 검출 정확도에 직결 → catastrophic failure mode (main 더럽힘) 회피 우선. v2.0.2 worktree 일반화 패턴 (LLM-judged) 과 정반대 방향 (백로그 명시).

### D-T3. Merge 전략 default — merge commit fixed

requirements D3 그대로. js-super 의 wave-parallel + per-task commit + RISK follow-up + 변경이력 footer 패턴과 일관성 ↑.

### D-T4. Step 1 dirty working tree 처리 — AskUserQuestion 2-choice

| 옵션 | 장점 | 단점 |
|---|---|---|
| **AskUserQuestion 2-choice** ← 채택 | 사용자 의도 보존 / 흐름 자연 | 2개 선택지 |
| 자동 commit 후 진행 | 흐름 매끄러움 | commit 메시지 임의 생성 → 사용자 의도 손상 |
| 무조건 차단 + 사용자 직접 commit 강제 | 안전 | 매번 사용자 메시지 입력 부담 |

### D-T5. 사후 처리 게이트 — multi-question batching (의미 결합)

worktree 제거 / 브랜치 삭제 / push 3건은 의미 결합 ("머지 완료 후 정리 단계") → AskUserQuestion `questions[]` 에 3개 묶음. v2.0.3 boilerplate 의 "multi-question batching 의미 결합 시 max 4" 룰 적용.

**비교**: 순차 3건 → 알람 3회 → 사용자 피로 ↑. 묶음 1회 → 알람 1회. 사용자 답변 효율 ↑.

### D-T6. parent worktree 명령 실행 방식 — `git -C <path>` (cd 없음)

| 옵션 | 장점 | 단점 |
|---|---|---|
| **`git -C <parent-path>`** ← 채택 | cwd 변경 X / skill 종료 후 사용자 자연스럽게 feature 워크트리 유지 / 사후 정리 흐름 자연 | 명령 1개당 `-C` 옵션 매번 |
| `cd <parent-path> && git ...` | 단순 | cwd 변경 → skill 종료 시점 cwd state 모호 / 사용자 혼란 |
| subprocess working dir 명시 | 안전 | bash 한정에서 복잡 |

### D-T7. 자동 충돌 해결 — 절대 X (백로그 § Non-goals)

## 6. 위험/사이드이펙트 (preliminary)

| R | 카테고리 | 위치 / 트리거 | Mitigation |
|---|---|---|---|
| R1 | breaking | Guard 검출 실패 시 main 워크트리에서 진행 | deterministic `git worktree list` + cwd 비교. 비표준 구조 시 명시 차단 (HARD-GATE). LLM-judged 회피 (D-T2). |
| R2 | side-effect | Step 3 자동 충돌 해결 시 데이터 손실 | 자동 해결 절대 X. 사용자 판단 강제 (AskUserQuestion). |
| R3 | side-effect | Step 4 가 Step 3 흡수 가정 깨질 때 (예: 사용자가 Step 3 중간 다른 commit 추가) | Step 4 진입 전 `git merge-base --is-ancestor origin/<parent> HEAD` 검증. 실패 시 차단. |
| R4 | breaking | force-push 실수 (백로그 § Non-goals) | 본문에 force 옵션 0 (push 만, --force 절대 X). Anti-Pattern 표 명시. |
| R5 | side-effect | 워크트리 force 제거 시 uncommitted 변경 손실 | normal remove 가 실패 시만 사용자 명시 force AskUserQuestion. default normal. |
| R6 | race | 동시에 다른 세션이 parent 워크트리 머지 | git 자체 lock 의존. skill 본문은 추가 lock 안 둠 (1인 개발 가정). |
| R7 | breaking | parent worktree 추론 실패 (multi-parent / nested) | Step 2 deterministic 검사 1줄. 비표준 구조 즉시 차단 (O1). |
| R8 | side-effect | `git merge --abort` 가 실패할 만큼 working tree 손상 | abort 실패 시 즉시 종료 + "수동 정리 필요" 안내. 자동 복구 X. |
