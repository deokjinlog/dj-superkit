# v2.0.4 — `worktree-merge-back` 신규 skill

> **Mode:** Socratic (auto-brainstorming 출력, free-form)
> 작성: 2026-05-16

## 한 줄 요약

dj-superkit 의 워크트리 기반 개발 흐름 종결부 (feature 워크트리 → parent main) 자동화. **"Merge down before merging up"** 패턴으로 충돌 해결은 feature sandbox 에서만 발생, main 워크트리는 항상 깨끗 유지. **worktree 안에서만 호출 가능** (main / 일반 작업트리에서 invoke 시 명시 차단).

## 배경

`setting-up-worktrees` 로 워크트리 자르고 작업 끝나면 사용자가 매번 수동으로:

1. 현재 워크트리에 commit
2. parent (main) 워크트리로 이동
3. `git merge <feature-branch>`
4. 충돌 시 → parent 쪽에서 해결 (위험: main 더럽힘)

→ **main 워크트리에서 충돌 해결 = main 깨질 위험.** 매번 사용자가 수동으로 신경. 1인/소수 개발 환경에서 자주 발생하는 반복 마찰.

## 핵심 결정

### D1. Skill 형태 — 신규 skill (`worktree-merge-back`) + slash command `/worktree-merge-back`

명시 invoke 만. 자동 발동 경로 없음 (`finishing-a-development-branch` 등 다른 skill 의 자동 호출 X — 사용자가 직접 결정).

**근거**: 워크트리 작업 종료 시점이 사용자 의도 명확한 분기점. 자동 발동 시 사용자가 "그냥 keep" 의도였을 때 잘못된 머지 발생 위험.

### D2. 워크트리 제약 — main / 일반 작업트리에서 invoke 시 명시 차단 (HARD-GATE)

```
❌ 이 skill 은 worktree 안에서만 사용 가능합니다.
   현재: <path> (main 워크트리 또는 일반 작업트리)
   먼저 `/worktree <feature-name>` 으로 워크트리 진입 후 다시 호출하세요.
```

**검출 방법** (deterministic):
- `git worktree list` 호출 후 현재 cwd 와 main worktree path 비교
- 현재 cwd == main worktree path → 차단
- `git rev-parse --is-inside-work-tree` 실패 → "git 저장소 아님" 차단

**근거**: 사용자가 v2.0.4 기획 시 강조한 핵심 안전성. 본 skill 의 전체 흐름이 "feature 워크트리 sandbox 에서 충돌 해결" 가정을 깔고 설계됨 → 가정 깨지면 main 깨질 위험.

### D3. Merge 전략 default — `merge commit` (`--no-ff`) 고정

**대안 비교**:

| 옵션 | 장점 | 단점 |
|---|---|---|
| **merge commit (--no-ff)** ← 채택 | history 추적 명확 / feature 묶음 보존 / revert 단순 (`git revert -m 1`) / Step 3 흡수 후라 conflict 없는 깨끗한 merge commit | 비선형 history (1인 개발 환경에서는 noise 가 아님) |
| squash | main history 선형 / 깔끔 | dj-superkit 의 multi-commit TDD 패턴 (task 단위 commit + RISK follow-up) 정보 손실. 변경이력 footer 와 mismatch |
| rebase | 선형 + feature commit 보존 | parent 가 비선형으로 변하면 충돌 가능성 ↑. Step 3 흡수 후의 깨끗한 merge 안전성 손상 |
| 매번 묻기 | 사용자 의도 완전 보존 | 흐름 마찰 ↑. dj-superkit 의 "흐름 안에서 진행" 철학 위배 |

**근거**: dj-superkit 의 wave-parallel + per-task commit + RISK follow-up + 변경이력 footer 패턴이 squash 와 충돌. merge commit 이 가장 일관성 ↑.

**사용자 override**: 사용자가 명시적으로 다른 전략을 요청하면 그 turn 한정 적용 (AskUserQuestion 으로 1회 확인). 본 skill 본문 default 변경 X.

### D4. Step 1 commit — working tree 변경 사항 있을 때만 AskUserQuestion

- working tree clean → 자동 Step 2 진행
- working tree 변경 있음 → AskUserQuestion `[지금 commit / 사용자가 직접 commit 후 재호출]`
  - "지금 commit" 선택 시 → 사용자가 메시지 입력 (free-form, 다음 turn)
  - "직접 commit 후 재호출" 선택 시 → skill 종료 + 안내문

**근거**: skill 이 commit 메시지를 임의로 생성하면 사용자 의도 손상. 사용자 메시지 입력 = 최소 마찰.

### D5. Step 3 — parent 흡수 (`git merge origin/<parent>`)

- `git fetch origin` (parent branch 최신화)
- `git merge origin/<parent-branch>` (default merge, rebase 안 함)
- **충돌 발생 시**:
  - 충돌 파일 리스트 노출
  - 메인 에이전트가 자동 해결 시도 **X** (사용자 판단)
  - 사용자에게 AskUserQuestion `[수동 해결 후 commit 했음 / abort]`
  - "abort" 선택 → `git merge --abort` 실행 후 skill 종료

**근거**: 자동 충돌 해결은 데이터 손실 위험 (한쪽 임의 채택). 사용자 판단 필수.

### D6. Step 4 — parent 워크트리로 머지

- `git -C <parent-worktree-path> merge --no-ff <feature-branch>` (D3 의 merge commit 전략)
- 이 시점 충돌 X 보장 (Step 3 흡수 완료)
- merge commit 메시지 AskUserQuestion `[기본 메시지 / 직접 입력]`
  - 기본 메시지 패턴: `Merge branch '<feature-branch>' into <parent-branch>`

**근거**: `git -C` 로 cd 없이 parent 워크트리에 직접 명령 실행 → 사용자 cwd 변경 X (skill 흐름 종료 후 사용자는 여전히 feature 워크트리에 있음 → 사후 정리 자연스러움).

### D7. Step 5 — 사후 처리 게이트 (모두 default `no`, 명시 yes 만)

3개 AskUserQuestion 순차 (또는 multi-question batching 으로 묶음 — 의미 결합 가능):

1. 워크트리 제거? (`git worktree remove <path>`) — default no
2. feature 브랜치 삭제? (`git branch -d <branch>`) — default no
3. parent push? (`git push origin <parent-branch>`) — default no

**근거**: 모두 destructive (worktree 제거 / 브랜치 삭제) 또는 원격 영향 (push) → 사용자 명시 yes 필수. 자동 적용 시 사용자가 "그대로 두기" 의도였을 때 자료 손실 위험.

### D8. 검증 fixture

`skills/worktree-merge-back/tests/` 에 3개 fixture (백로그 § 검증):

1. **H14-normal** — 워크트리 안 + 충돌 없음 정상 1회 완주
2. **H15-conflict** — Step 3 에서 main 쪽 변경 + feature 쪽 변경 동일 파일 → 충돌 → 사용자 수동 해결 → 정상 진행
3. **H16-guard** — main 워크트리에서 invoke 시 차단 메시지 + skill 종료

각 README.md 만 (Python 코드 없음, 메타 dogfood 패턴).

## 우려 / 해결

### O1. 사용자가 fork-tracking 워크트리에서 호출 시 parent 추론 실패

`git worktree list` 결과가 비어있거나 multi-parent 경우. → 명시 차단 + 안내문:

```
❌ parent 워크트리 추론 실패. 현재 워크트리 구조가 예상과 다름.
   `git worktree list` 결과를 확인하고 수동으로 머지하세요.
```

**Mitigation**: Step 2 안에 deterministic 검사 1줄 (`git worktree list | wc -l`). 비표준 구조 즉시 차단.

### O2. Step 3 머지 중간 abort 시 rollback

- `git merge --abort` 가능한 상태인지 확인 (`git merge --abort` 실패 시 working tree 손상 의심)
- abort 후 skill 종료 + "수동 정리 필요" 안내

**Mitigation**: abort 도 사용자 명시 (AskUserQuestion). 자동 abort X.

### O3. 사용자가 흐름 중간 interrupt (Ctrl+C / 다른 명령)

각 Step 별 실패 시 즉시 중단 — 다음 단계 자동 진행 X. 사용자가 수동으로 정리.

**Mitigation**: 각 Step 끝마다 git 상태 점검 → 다음 Step 진입 전 확인. abort 흐름 명시.

### O4. 워크트리 제거 시 uncommitted 변경 있으면 force 여부

- `git worktree remove <path>` (force 없음) — uncommitted 변경 있으면 실패
- 실패 시 사용자에게 안내 + force 여부 AskUserQuestion `[force 제거 / 그대로 두기]`
- "force 제거" 선택 시 `git worktree remove --force <path>` 실행

**Mitigation**: force 도 사용자 명시. default 는 안전한 normal remove.

### O5. parent push 시 conflict (remote 가 앞서있음)

- normal `git push` (force 안 함)
- conflict 시 안내 + skill 종료. 사용자가 수동 `git pull --rebase` 후 재시도.

**Mitigation**: force-push 절대 X (백로그 § Non-goals).

### O6. v2.0.3 AskUserQuestion 강제 룰 적용

본 skill 의 모든 사용자 질문 → AskUserQuestion. 산문 질문 0건.

**Mitigation**: skill body 상단 "사용자 질문 룰 (v2.0.3+)" boilerplate 자동 박음 (8 skill 룰의 9번째 적용).
