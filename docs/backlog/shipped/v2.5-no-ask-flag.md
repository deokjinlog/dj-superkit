# v2.5 후보 — `--no-ask` 플래그 도입 (AskUserQuestion 도구 우회)

> 작성: 2026-05-23
> 우선순위: 중간 — AskUserQuestion 도구의 응답 지연 / 불안정성으로 사용자 마찰. 8 skill boilerplate 옵트인 플래그로 우회.

## 문제

v2.0.3+ 에서 8 skill body 에 박힌 boilerplate (v2.0.3+ "사용자 질문 룰" + v2.3.5+ "AskUserQuestion 도구 우선") 가 모든 사용자 질문을 `AskUserQuestion` 도구로 강제. 의도는 알람 누락 방지 (Notification.elicitation_dialog 매처 → repeat-alert.sh 발화).

하지만 최근 `AskUserQuestion` 도구 자체가:

- 응답이 느림 (도구 호출 → 사용자 응답 → 메인 turn 복귀 latency ↑)
- 가끔 불안정 (도구 호출이 fail 하거나 응답 자체가 안 돌아오는 케이스 신고)

→ 게이트 답변마다 답답함. 4 단계 게이트 + 흐름 중간 clarifying Q 모두 누적되면 사용자 경험 저해.

실사용 시나리오 (사용자 신고):

> "요즘 AskUseQuestion 이게 꽤 불안정해 막 대답이 엄청 느려.. 그래서 그.. 예를들어 /brainstorming --no-ask 하면 그 AskUseQuestion 도구를 절대 하지않는 버전을 만들고 싶어"

## 해결 — `--no-ask` 옵트인 플래그

### 룰 (요약)

8 skill body 의 boilerplate 에 `--no-ask` 플래그 분기 추가. 이 플래그가 명시 invoke 시:

- **일반 4 skill** (brainstorming / designing-direction / writing-plans / executing-plans): 모든 사용자 질문을 **prose** (메인 turn 자유 텍스트, 채팅 메시지 형식) 로만 처리. `AskUserQuestion` 도구 호출 0. 게이트는 살아 있음 — 사용자 prose 응답 기다림.
- **auto-* 4 skill** (auto-brainstorming / auto-designing-direction / auto-writing-plans / auto-executing-plans): **동일** — clarifying Q 자체는 그대로 prose 로 묻되, `AskUserQuestion` 도구 호출 0 보장. 내부 escalation 경로 (BLOCKED 자가복구 실패 / critical 7 케이스 재질문 / Other 모호 응답 재질문) 에서도 도구 호출 X.

### 적용 범위 (8 skill)

| Skill | 기본 동작 | `--no-ask` 진입 시 |
|---|---|---|
| brainstorming | AskUserQuestion 도구로 모든 질문 | prose 질문만, 도구 호출 X |
| designing-direction | 동일 | 동일 |
| writing-plans | 동일 | 동일 |
| executing-plans | 동일 | 동일 |
| auto-brainstorming | clarifying Q prose / 자동 진행 / 내부 escalation 도구 호출 가능 | clarifying Q prose 유지 + 도구 호출 0 보장 (escalation 포함) |
| auto-designing-direction | 동일 | 동일 |
| auto-writing-plans | 동일 | 동일 |
| auto-executing-plans | 동일 | 동일 |

### 적용 범위 외 (v2.5 1차 release 제외)

- `fast-tasks` — v2.5.x 후속 후보 (Socratic 2~3 Q × N 형식 자체가 `--no-ask` 와 충돌 — clarifying Q 가 본질 기능)
- `worktree-merge-back` — destructive 작업 후처리 default yes 안 됨. 사용자 게이트 본질. `--no-ask` 비적용
- og-* — upstream mirror 보존. 변경 X

### 플래그 이름

`--no-ask` 채택. 짧고 의미 명확. 영어 식별자 보존 룰 따름.

대안 (`--prose-only`, `--quick`, `--silent`) 보다 사용자 신고 표현 (`/brainstorming --no-ask`) 와 일치 — 사용자 인지 비용 최저.

### 진입 형식

```
/brainstorm <slug> --no-ask
/auto-brainstorm <slug> --no-ask
/design <slug> --no-ask
/auto-design <slug> --no-ask
/write-plan <slug> --no-ask
/auto-write-plan <slug> --no-ask
/execute-plan <slug> --no-ask
/auto-execute-plan <slug> --no-ask
```

플래그 위치 자유 (slug 앞/뒤 모두 허용 — argument parser 강건성).

### CLAUDE.md 분기 무효

CLAUDE.md 의 v2.3.5+ "AskUserQuestion 도구 우선" 글로벌 룰은 **contributor 메모** — 사용자 환경에 안 전달됨. 따라서:

- CLAUDE.md 변경은 v2.5 핵심 작업 아님
- 진짜 사용자 영향 위치는 **8 skill body 의 boilerplate** (v2.0.3+ "사용자 질문 룰" + v2.3.5+ "AskUserQuestion 도구 우선" 섹션)
- 그 boilerplate 에 `--no-ask` 분기 추가가 v2.5 의 실제 작업

사용자가 `--no-ask` 플래그로 명시 invoke 한 경우:

## 미정 사항

- v2.3.5+ critical/non-critical 판정 룰과의 상호작용 — `--no-ask` 진입 시 critical case 도 prose 로 처리? 아니면 critical case 만 예외적으로 도구 호출 허용? (default: 모든 case prose — 사용자 의도 일관성 우선)
- 알람 fire X 안내 — turn 시작 시 "AskUserQuestion 호출 X 모드 — 응답 알람 없음, 백그라운드 작업 중이면 직접 체크" 한 줄 boilerplate 박을지?
- 플래그 persistence — 한 turn 만? 세션 전체? 아니면 매 슬래시 invoke 마다 명시? (default: 매 invoke 마다 명시 — 사용자 통제권 우선)
- 8 skill body 외 흐름 중간에 메인 turn 이 ad-hoc 결정 요청하는 경우 — `--no-ask` 컨텍스트를 메인이 인지하고 prose 로 전환 가능한지 (기술 검증 필요)
