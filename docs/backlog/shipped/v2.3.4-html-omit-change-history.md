# A. 변경이력 UI 생략 허용

## 문제

현재 `html-companion-prompt.md` 룰: `.md` 의 모든 섹션을 `.html` 에 1:1 매핑. 즉 `## 변경이력` footer 도 `.html` 안에 렌더링됨.

그러나 변경이력 footer 는:
- AI 가 추적용으로 쓰는 **audit log** (CH-id / 이유 / 영향범위)
- `.html` 은 사람용 빠른 훑기 surface
- 사람이 `.html` 에서 변경이력 entries 를 읽을 일이 거의 없음 (보통 git log / change-history skill 로 확인)
- 시각적으로 길고 hierarchy 평탄 — `.html` 의 핵심 가독성을 깎음

→ `.html` 에서 변경이력 UI 는 **선택적으로 생략 가능** 으로 룰 완화.

## 해결 — `html-companion-prompt.md` 본문 룰 1개 추가

`## 변경이력` 섹션 (또는 동등 footer) 은 `.html` 에서 **생략 OK** (사람 가독성 우선). 단:

- 생략 시 `.html` 푸터 한 줄 안내: "변경이력은 `.md` 원본 참조" (또는 동등 문구)
- 생략 vs 노출은 subagent B 의 자율 판단 (정보 hierarchy 기준)

### 기존 의미 1:1 보존 룰과의 관계

- 본 룰은 **변경이력 footer 한정 예외**
- FR / AC / 배경 / 결정 / NFR / Non-goals / 미정 등 본문 콘텐츠는 그대로 1:1 보존 (v2.3.3 의 primary visibility 룰과 정합)
- v2.3.3 의 "secondary content 만 progressive disclosure" 룰에서 변경이력은 secondary 범주였음 — 이번 룰은 한 발 더 나아가 **생략 자체** 허용

## Anti-Patterns (A 항목)

| 안티 패턴 | 이유 |
|---|---|
| 변경이력 entry 본문을 `.html` 에 그대로 1:1 렌더 (강제) | 사람 가독성 ↓. 본 룰로 생략 허용. |
| 변경이력 footer 만 의미 변형 / 요약 | 생략은 OK지만 요약 변형은 X. 의미 1:1 룰 보존. |
| 변경이력 외 본문 콘텐츠도 같이 생략 | 본 룰은 변경이력 footer 한정. FR / AC 등 본문 생략은 v2.3.3 룰 위반. |

# B. 다크 프리미엄 default + 4-요소 옵셔널

## 문제 (사용자 신고)

`/Users/goldenplanet/Desktop/sa-dashboard-overhaul-tech-design.html` 검토 시:

> "지금 배경의 흰색인데.. 그 글에 회색깔로 칠해진 부분때문에 정말 사람이 알아보기 힘들 지경이야"

신고된 html 의 스타일 변수 발췌:
```css
:root {
  --bg: #f8f9fa;
  --text: #212529;
  --text-muted: #6c757d;   /* ← 한글 본문 muted, 가독성 ↓ */
}
@media (prefers-color-scheme: dark) {
  :root { --bg: #0d1117; ... }   /* dark 분기는 OS 설정 의존 */
}
```

→ default 가 light 모드. `#6c757d` (회색) on `#f8f9fa` (거의 흰색) 의 한글 가독성이 특히 떨어짐. 사용자 OS 가 light 면 회색 글자가 그대로 노출.

## 회귀 원인 — v2.2.4 톤 자유 룰의 부작용

v2.2.4 에서:
- 6 톤 inspiration 자유 선택 (docs portal / dashboard / **editorial** / playful / brutalist / experimental)
- "variety = feature" — 매 호출 다르게

→ subagent B 가 **editorial / docs portal** 톤을 골라 흰 배경 + 회색 muted 본문 패턴을 적용하는 경우 발생. 디자인 다양성은 가치 있으나 **가독성 baseline 이 깨지면 무용지물**.

## 해결 — 다크 프리미엄 default + 4-요소 옵셔널

### 룰 B-1: 비주얼 테마 = "다크 프리미엄" default

**모든** generating-html 산출물의 default 테마:

- **배경 (dark base)**: 깊은 어두운 톤 — `#0a0b0f` / `#0d1117` / `#111218` 류
- **텍스트 (high contrast)**: 밝은 톤 — `#e6edf3` / `#f0f6fc` / `#e8eaf0`. **muted 도 contrast 충분히** (`#8b949e` 보다 어둡지 않게)
- **악센트 (premium)**: 보라/시안/핑크 그라데이션 또는 단색 강조 — 차분하면서 시각 임팩트
- **카드 / 표면**: 반투명 흰색 overlay (`rgba(255,255,255,0.04~0.07)`) — 깊이감
- **그림자 / glow**: subtle drop shadow + accent glow (`rgba(124,109,255,0.25)` 류)
- **타이포**: heading bold, body 1.6~1.7 line-height, 한글 친화 폰트 스택

**다크 프리미엄 = "Premium dashboard / 고급 분석 도구" 톤**. 사무 / 회의록 / 흰 배경 editorial 톤 금지.

라이트 모드는 **`prefers-color-scheme: light`** 분기로만 제공 (default 가 dark, 명시적 라이트 선택은 OS 설정 따라감).

### 룰 B-2: 4-요소 옵셔널 — LLM 자율 판단

산출물에 포함 **가능** 한 요소 (모두 옵셔널):

| 요소 | 적용 케이스 (예시) |
|---|---|
| **데이터 차트/그래프** | 숫자 비교 / 분포 / 시계열 / 비율 데이터가 본문에 있을 때 |
| **인터랙티브 요소** | TOC / dark-mode toggle / copy 버튼 / smooth scroll / 보조 disclosure (primary 본문 X) |
| **다이어그램 / 플로우** | 아키텍처 / 데이터 흐름 / DAG / 컴포넌트 관계 |
| **핵심 요약 카드** | FR / 결정 / 위험 등 묶음 형태로 일관 표현 가능할 때 |

**룰**: 본문 콘텐츠가 위 요소에 맞지 않으면 **사용하지 마라**. 4 요소 모두 적용해야 한다는 강제 X. LLM 이 콘텐츠 보고 판단.

### 룰 B-3: v2.3.3 룰과의 정합

v2.3.3 의 "primary content default-visible" 룰 그대로:
- FR / AC / 배경 / 결정 등 primary 콘텐츠는 default-visible
- 인터랙티브 요소는 secondary 에만 (TOC / 변경이력 / dark-mode toggle 등)

## Anti-Patterns (B 항목)

| 안티 패턴 | 이유 |
|---|---|
| Default 흰 배경 (`#fff` / `#f8f9fa` 류) + 회색 muted 본문 | 신고 사례 직격. 한글 가독성 망가짐. |
| `prefers-color-scheme: dark` 만 dark 분기 (default light) | OS 설정 의존성 위험. default 가 dark 여야 함. |
| 6 톤 (editorial / playful / brutalist 등) 자유 선택 | v2.2.4 룰 약화. 다크 프리미엄 baseline 강제. |
| 4 요소 모두 의무 사용 (차트 / 인터랙티브 / 다이어그램 / 카드) | 4 요소는 옵셔널. 콘텐츠와 안 맞으면 사용 X. |
| Muted 텍스트가 contrast 4.5:1 미만 (WCAG AA) | 가독성 baseline. dark 배경에서도 muted 는 어두운 회색 X. |

## 디자인 톤 좁힘 — v2.2.4 룰 보정

v2.2.4 의 6 톤 inspiration → **다크 프리미엄 1 톤 default**.

단 다크 프리미엄 내에서 variety 는 보존:
- 악센트 색상 팔레트 (보라+시안 / 시안+핑크 / 청록+에메랄드 등 매번 다르게)
- 카드 / overlay / glow 강도 (subtle ↔ pronounced)
- 헤더 typography / spacing
- 다이어그램 스타일 (flat / glassmorphism / neon outline 등)

"variety = feature" 정신은 유지, baseline 톤만 통일.

# C. impl-plan 의 원본/수정본 diff 시각화

## 문제

`impl-plan.md` 의 task 블록은 v2.0.0 byte-copy 패턴 이후 항상 두 코드 블록을 같이 가짐:

```markdown
### Task 1
**Model**: haiku

**원본**:
\`\`\`python
def foo():
    return 1
\`\`\`

**수정본**:
\`\`\`python
def foo():
    return 2  # ← 이 줄만 변경
\`\`\`
```

현재 `.html` 생성 시 두 블록을 **별개 코드 블록 두 개로 그대로 렌더링**. 사람이 "뭐가 바뀌었지?" 비교를 라인별로 직접 해야 함:

- 코드 50+ 줄에서 5 라인만 변경된 경우 → 두 블록 시각 diff 부담 큼
- impl-plan 검토 효율 ↓ — 사용자가 plan 의 변경 의도 빠르게 catch 못 함
- 신고 사례는 없지만 byte-copy 룰 (v2.0.0+) 정착 이후 자연스럽게 누적된 UX 부채

## 회귀 원인 — 아님 (신규 개선)

v2.0.0 의 byte-copy 룰 (`**원본**` + `**수정본**` 두 블록) 자체는 plan 정확성에 필수. 다만 그 산출물을 `.html` 로 시각화하는 단계에서 **diff hint 가 빠져 있어** 사람 검토 surface 가 부족함.

## 해결 — `html-companion-prompt.md` 룰 1개 추가

`impl-plan` 류 산출물에서 `**원본**` + `**수정본**` 쌍 (또는 동등한 before/after 코드 블록 쌍) 을 감지하면, `.html` 에서 **unified diff 시각화** 로 렌더:

### 룰 C-1: diff 시각화 강제

| 라인 종류 | 시각 표현 |
|---|---|
| **추가 라인** (수정본에만 존재) | 초록 배경 (`rgba(46,160,67,0.15)` 류) + 좌측 `+` 마커 + 초록 텍스트 톤 |
| **삭제 라인** (원본에만 존재) | 빨강 배경 (`rgba(248,81,73,0.15)` 류) + 좌측 `-` 마커 + 빨강 텍스트 톤 |
| **변경 X 라인** (양쪽 동일) | context — 기본 톤, 좌측 공백 마커 |

- 다크 프리미엄 default (룰 B-1) 위에서 적용 → 초록/빨강은 dark 배경에 어울리는 톤 사용 (GitHub dark 류)
- WCAG AA contrast 4.5:1 보존 (룰 B Anti-Pattern 과 정합)

### 룰 C-2: context 라인 진보적 공개 OK

원본 50줄 / 변경 5줄 케이스에서:
- 변경 라인 ± N (기본 3) 만 default-visible
- 그 외 unchanged 영역은 "... unchanged N lines ..." 형태로 collapse OK (`<details>` 사용 OK)
- 이는 v2.3.3 "primary content default-visible" 룰의 **예외** (코드 raw 는 secondary 성격) — 단 변경 라인 자체는 항상 default-visible

### 룰 C-3: raw 원본/수정본 병행 옵셔널

LLM 자율 판단으로 두 raw 블록을 collapse 형태로 같이 제공 OK (사용자가 unified diff 외 원본/수정본 그대로 보고 싶을 때). 단:
- raw 블록 default-collapsed (`<details>`) OK
- diff 시각화가 primary, raw 는 보조

### 룰 C-4: 신규 파일 / 삭제 파일 fallback

- 신규 파일 (원본 X / 수정본 only) → 단순 코드 블록 (초록 배경 전체 적용 OK, diff 형식 강요 X)
- 삭제 파일 (원본 only / 수정본 X) → 단순 코드 블록 (빨강 배경 전체 적용 OK)
- 즉 진짜 before/after 쌍일 때만 unified diff 시각화

## Anti-Patterns (C 항목)

| 안티 패턴 | 이유 |
|---|---|
| 원본 / 수정본 두 블록 별개 렌더 (diff 시각화 X) | 신규 룰 위반. 사람 검토 효율 ↓. |
| diff 시각화에 외부 라이브러리 의존 (e.g. diff2html CDN) | self-contained 룰 위반. inline CSS / SVG 만. |
| word-level / character-level diff 도 함께 적용 (LLM 자율 구현) | 의역/오류 위험. line-level 단순 diff 만. |
| semantic diff (의미 변경 요약 / hint 텍스트) | 의미 1:1 룰 위반. 변경 의도 해석은 plan 본문에 맡김. |
| diff 라인 색상이 dark 배경에서 contrast 4.5:1 미만 | 룰 B (WCAG AA) 와 충돌. |
| context 라인 collapse 강제 (전부 hide) | 변경 라인 ± N 은 default-visible 보장. |
| 신규/삭제 파일도 억지로 unified diff 형식 | C-4 fallback 위반. 단순 코드 블록 OK. |

## 정합

- v2.0.0 byte-copy 룰 보존 — plan 의 두 블록은 그대로 (`.md` 원본 무변형)
- v2.3.3 primary visibility 룰 보존 — 변경 라인은 default-visible, unchanged 만 collapse OK
- 룰 B 다크 프리미엄 default 위에서 초록/빨강 톤 적용
- writing-plans / executing-plans / js-super-sub-driven skill body 변경 X — `.html` 렌더링 단계 룰만

- raw 원본/수정본 블록 병행 제공 여부 확인 (LLM 자율, default-collapsed OK)
- `.md` 원본의 `**원본**` + `**수정본**` 블록은 무변형 (byte-copy 룰 보존)

## Non-goals (이번 버전 안 함)

- 변경이력 자동 요약 (생략만 허용, 요약은 의미 변형 위험)
- `.md` 에서 변경이력 제거 (`.html` 만 룰 변경, `.md` 는 그대로)
- 변경이력 외 footer 류 (e.g. metadata) 의 생략 룰 확장
- 흰 배경 / 라이트 모드 완전 금지 — `prefers-color-scheme: light` 분기는 OS 선호 따라 제공 OK (default 만 dark)
- 6 톤 inspiration 자체 삭제 — 다크 프리미엄 안에서 variety 보존
- 차트 라이브러리 도입 (inline SVG / CSS 만, self-contained 보존)
- 모바일 반응형 강제
- (C) word-level / character-level diff — line-level 만
- (C) semantic diff / 변경 의도 요약 텍스트 — 의역 위험
- (C) 3-way diff / inline 편집 / Apply 버튼 — `.html` 은 read-only surface
- (C) diff 알고리즘 정교화 (move detection 등) — 단순 라인 비교만

## 미정 사항

- 푸터 안내 정확한 문구 ("변경이력은 .md 원본 참조" 가 default?)
- entry 개수 임계 (예: 5+ 면 생략 권장, 미만이면 노출?) — 룰 강제 X, subagent 자율
- "다크 프리미엄" 의 정확한 팔레트 룰 — 예시만 제공 vs 명시 표준 (저자 의견: 예시만, LLM 자율)
- WCAG AA 4.5:1 자동 검증 도구 도입 여부 (subagent self-check 단계에 추가?)
- 4 요소 외 추가 옵셔널 (예: 비디오 / 인용 / footnote) 룰 — 미래 후보
- 사용자가 명시적으로 "라이트 톤" 원할 때 override 방법 (`/sync-html --theme=light`?) — v2.3.x 후속 후보
- (C) diff 표현 형식 — unified 만 vs side-by-side 옵션도 제공? (저자 의견: unified 만, side-by-side 는 mobile/narrow viewport 깨짐)
- (C) context 라인 ± N 기본값 — 3 vs 5 vs LLM 자율?
- (C) raw 원본/수정본 병행 제공 default — collapse vs hide entirely?
- (C) 다중 hunk (한 task 안 코드 블록 떨어진 위치 2+ 곳 변경) 표현 — 한 diff 안에 hunk 분리 표시 vs 별개 diff 블록?
- (C) impl-plan 외 다른 산출물 (req / tech-design) 에 before/after 코드 블록 쌍이 있다면 같은 룰 적용? (현재 룰은 impl-plan 한정)
