# A. execute-plan 실행 중 non-critical mid-flight 재질문 지양

## 문제 (사용자 신고)

impl-plan 작성 완료 후 `/execute-plan` (inline) 또는 `js-super-sub-driven` (subagent) 진입 시:

1. 사용자가 **모드 선택 게이트** 에서 inline / subagent 중 하나 명시 선택
2. 사용자는 모드 선택 후 백그라운드로 두고 **다른 작업으로 이동** (회의 / 다른 코드 작업 / 휴식)
3. 메인 에이전트가 실행 중간에 **mode 변경 / 분기 결정** 을 사용자에게 되묻는 경우 발생

### 신고된 패턴 (사용자 원문)

> "예를들어 멈춰서 지금 inline이지만 (T2까지는 진행한상태) T3~T5는 병렬 작업할건데 괜찮나요? 라고 되묻는 구간이있어"
>
> "사용자는 sub-agent인지 inline인지 고르면 신경 안쓴채로 다른 작업 업무볼탠데 이런 경우의 되묻는건 좀 지양하는 방식으로 했으면 좋겠어"
>
> "크리티컬한 문제일 경우에는 되묻는게 맞아"

### 핵심 회귀

- **모드 선택 = 사용자 위임 신호**. 사용자는 "알아서 진행해라" 의 의미로 모드를 고른 것
- 실행 중 자잘한 최적화 결정 (병렬 / 묶음 / 순서 조정) 까지 재질문 → 사용자 attention 강제 회수
- 사용자가 자리에 없으면 실행 정지 → 진행률 0
- 알람 시스템 (v1.1.x+ `repeat-alert.sh` 4-layer) 이 fire → OS 알람 강제 → 백그라운드 작업 방해

## 회귀 원인 (추정)

executing-plans / js-super-sub-driven 본문에 박힌 안전성 룰들이 **과보호** 로 작동:

- "자동승인 절대 X" (v1.1.12+) — 모든 사용자 영향 결정에 게이트 강제
- AskUserQuestion 강제 룰 (v2.0.3+) — 8 skill 에 일관 적용
- subagent mode 의 W-2 분기 (BLOCKED → reorder dispatch) — 안전성 우선 설계

→ 안전성은 보존하되 **non-critical 결정 자체를 안 만드는 게** 정답.

## 해결 — "Critical / Non-critical 판정 룰" 추가

### 룰 1: 크리티컬 결정만 사용자 재질문 (mandatory)

다음 케이스는 **반드시 사용자 재질문** (AskUserQuestion):

| 케이스 | 이유 |
|---|---|
| 사용자가 선택한 모드 자체를 **변경** 해야 할 때 (inline → subagent / 반대) | 약속 위반. 명시 동의 필수. |
| plan 의 task 범위를 **확장** (계획 안된 파일 / 함수 손대야 함) | scope creep — 사용자 의도 모호 |
| **파괴적 작업** (rm -rf / git reset --hard / force-push / 데이터 손실 위험) | 비가역 |
| plan 안 task 간 **충돌 발견** (task A 수정본이 task B 원본과 불일치) | byte-copy 룰 위반, plan 재작성 필요 |
| BLOCKED 보고 후 **reorder 단계도 자동 복구 실패** | 사용자 직접 개입 필요 |
| **외부 서비스 호출** (push / PR 생성 / 외부 API 트리거) | blast radius 커짐 |
| 사용자가 명시 약속 X 한 **새 의존성 / 외부 도구** 도입 | 약속 외 변경 |

### 룰 2: Non-critical 최적화는 자율 진행 (no-ask)

다음 케이스는 **재질문 없이 자율 결정**:

| 케이스 | 자율 결정 방향 |
|---|---|
| **task 병렬 vs 순차** 실행 여부 | plan 의 dependencies 만족 시 병렬 default (성능 우선) |
| **task 묶음** (같은 파일 mechanical 변경 3-AND 룰 만족 시) | 묶음 default (v2.0.1+ 룰) |
| **task 안 보조 결정** (변수명 / format / order of imports) | plan 의 `**원본**` + `**수정본**` byte-copy 우선, 없으면 LLM 자율 |
| **dispatch model 선택** (haiku / sonnet) | plan 의 `**Model**:` 필드 우선, 없으면 기본 룰 |
| **task 완료 후 다음 task 진입** 타이밍 | 자동 진입 (게이트 X) |
| **중간 결과 보고 빈도** | 매 task X, 매 wave (3-5 task) 단위 OR BLOCKED 시만 |

### 룰 3: 모드 선택 = 사용자 위임 신호 (의미 명시)

executing-plans / js-super-sub-driven 본문에 다음 의미 박기:

> "사용자가 inline / subagent 모드를 선택한 시점부터, 그 모드에 내포된 진행 방식 (병렬 / 묶음 / 자동 진입) 은 묵시 동의로 간주한다. 사용자는 모드 선택 후 백그라운드 작업으로 이동할 수 있어야 한다. **모드 진행 중 추가 게이트는 룰 1 (크리티컬) 에 해당하지 않으면 차단**."

### 룰 4: BLOCKED 자가 복구 우선

subagent mode 의 W-2 분기 (v2.0.0+):
- implementer BLOCKED 보고 → **자동으로 reorder dispatch** (사용자 개입 X)
- reorder 도 실패 시에만 → 사용자 재질문 (룰 1 의 마지막 케이스)

inline mode 동등:
- task 실행 중 의도 모호 발견 시 → **plan 재독 후 self-correct 시도**
- self-correct 실패 시에만 → 사용자 재질문

## Anti-Patterns

| 안티 패턴 | 이유 |
|---|---|
| "T3~T5 병렬로 진행해도 될까요?" 류 게이트 | 룰 2 위반. plan dependencies 만족 시 자율 진행. |
| 매 task 완료 후 "다음 task 진입할까요?" 게이트 | 룰 3 위반. 모드 선택 = 진행 위임. |
| "같은 파일이라 묶을까요?" 게이트 | 룰 2 위반. 3-AND 룰 (v2.0.1+) 으로 자동 판정. |
| BLOCKED → 곧장 사용자 재질문 (reorder skip) | 룰 4 위반. reorder dispatch 자동 시도 우선. |
| dispatch model 변경 시 게이트 | 룰 2 위반. plan 의 `**Model**:` 필드 우선. |
| 변수명 / format / import 순서 게이트 | 룰 2 위반. plan 의 byte-copy 우선, 없으면 자율. |
| 사용자 모드 선택 무시하고 inline → subagent 자동 전환 | 룰 1 위반. 모드 변경은 명시 동의 필수. |
| 모든 mid-flight 결정을 "안전성" 명목으로 게이트 | 과보호. 룰 1 7 케이스 외엔 자율. |

# B. 사용자 질문 시 AskUserQuestion 도구 사용 강제

## 문제 (사용자 신고)

> "어느순간부터 AskUseQuestion이 잘 실행이 안될때가 있어.. 가급적이면 질문할떄 이걸 써야하는데 흠.."

회귀 패턴 — 메인 에이전트가 사용자에게 결정을 요청해야 할 때 **AskUserQuestion 도구 호출** 대신 **prose 자연어 응답** 으로 질문하는 경우:

- "이렇게 진행할까요?" 류 자연어 질문 → LLM 응답 stream 으로 출력
- AskUserQuestion 도구 미호출 → `Notification.elicitation_dialog` 이벤트 발화 X
- 알람 시스템 (`repeat-alert.sh` 4-layer, v1.1.x+) fire X
- 사용자가 백그라운드 작업 중일 때 질문 catch 못 함 → 응답 멈춤

### 영향

- 사용자 attention 손실 — 알람 안 옴
- AskUserQuestion 의 다른 기능 손실:
  - 옵션 다중 선택 (multiSelect)
  - 헤더 chip / description 구조화
  - preview (mockup / code snippet 비교)
  - "Other" 자유 응답 / 모호 응답 처리 (v2.1.1+)
- 일관성 손실 — 같은 종류 질문이 어떤 흐름에선 도구, 다른 흐름에선 prose

## 회귀 원인 (추정)

기존 룰 (v2.0.3+) 은 **8 skill body 안 boilerplate** 형태:
- `brainstorming` / `designing-direction` / `writing-plans` / `executing-plans` + auto-* 4개

이 boilerplate 가 박힌 흐름 안에서는 AskUserQuestion 호출 일관 발화. 그러나:
- 메인 turn 의 **일반 ad-hoc 결정** (skill body 안 아님)
- skill body 안인데 boilerplate 가 명시 안 한 결정 시점
- v2.3.5 룰 1 (critical 7 케이스) 재질문 시점 — 본 룰이 명시 안 하면 prose fallback

→ LLM 의 default 응답 양식이 prose 라서 명시 강제 없으면 도구 호출 skip.

## 해결 — 전역 "사용자 질문 = AskUserQuestion 도구" 룰

### 룰 B-1: 사용자 결정 요청 = AskUserQuestion 도구 호출 (default)

메인 에이전트가 사용자에게 **결정 / 선택 / 동의** 를 요청하는 모든 경우 → **AskUserQuestion 도구 호출** default.

대상 흐름 (skill body 무관 전역 적용):
- 모든 skill body 안 게이트 (기존 v2.0.3+ 룰)
- 메인 turn 의 ad-hoc 결정 요청
- v2.3.5 룰 1 (critical 7 케이스) 재질문 시점
- 사용자가 모호 응답 시 재질문 (v2.1.1+ Other 룰)
- 모드 선택 게이트 진입 시점
- BLOCKED → reorder 도 실패 시 사용자 개입 요청

### 룰 B-2: prose 질문 허용 예외 (좁게)

다음 경우에만 prose 자연어 질문 OK:
- 사용자가 명시적으로 **자유 텍스트 / 긴 응답** 을 요구하는 경우 (예: "어떻게 생각해?" 류 brainstorming open question)
- 사용자가 이미 응답한 직후 **확인용 단순 ack** (예: "알겠어요" / "맞아요?") — 단 이것도 AskUserQuestion (yes/no 옵션) 권장
- 질문이 아닌 **상태 보고 / 진행 알림** (질문 형식 아님, 단순 ".. 진행 중입니다" / ".. 완료했습니다")

→ "사용자가 결정을 골라야 하는 모든 질문" 은 무조건 AskUserQuestion.

### 룰 B-3: AskUserQuestion 호출 시 알람 fire 보장

- AskUserQuestion 호출 → `Notification.elicitation_dialog` 발화 (Claude Code 기본 동작)
- `~/.claude/settings.json` 의 elicitation_dialog 매처 → `repeat-alert.sh` fire
- 사용자가 백그라운드 작업 중일 때 OS 알람 캐치 → 응답 흐름 보존

→ 본 룰은 알람 시스템 (v1.1.x+) 과 결합. 알람 일관성 = 도구 호출 일관성.

### 룰 B-4: 메인 에이전트 system instruction 강화

skill body 외부 영역의 룰이라 **2 위치 모두 박기**:

1. **각 skill body 안** — 기존 8 skill boilerplate (v2.0.3+) 보존 + 본 룰 명시 (cross-reference)
2. **CLAUDE.md 본문 안** (프로젝트 글로벌) — "사용자 질문은 AskUserQuestion 도구 우선" 룰 명시 → 메인 turn ad-hoc 결정에도 적용

## Anti-Patterns (B 항목)

| 안티 패턴 | 이유 |
|---|---|
| "이렇게 진행할까요?" 류 prose 자연어 질문 | 룰 B-1 위반. AskUserQuestion (yes/no 옵션) 사용 강제. |
| "옵션 1: ... 옵션 2: ... 어느 쪽?" prose 멀티 옵션 | 룰 B-1 위반. AskUserQuestion options 필드 사용. |
| 마크다운 체크박스 / numbered list 로 사용자 선택 유도 (prose) | 룰 B-1 위반. AskUserQuestion 사용. |
| v2.3.5 룰 1 critical 재질문을 prose 로 ("force-push 해도 될까요?") | 룰 B-1 위반. critical 일수록 AskUserQuestion + 알람 fire 필수. |
| AskUserQuestion 호출 직후 prose 로 추가 질문 (이중 질문) | 한 turn 한 도구 호출 / 답변 흐름 보존. |
