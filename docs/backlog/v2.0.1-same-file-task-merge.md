# v2.0.1 same-file mechanical task — plan 작성 시 1 task 묶음 룰

> **Version note (v2.0.1)**: writing-plans / auto-writing-plans 가 같은 파일만 만지는 mechanical task 들을 과분해. 결과: wave-parallel 가치 0 (file overlap → 직렬 wave 강제) + commit fragmentation. minor patch (skill 본문 룰 추가) 이라 v2.0.1 적합. 자동 merge helper (Python) 는 후속 검토 — 본 release 미포함.

## 한 줄 요약

writing-plans / auto-writing-plans 의 bite-sized TDD 룰이 "1 logical change = 1 task" 를 너무 엄격히 적용 → 같은 파일의 작은 mechanical 변경들 (예: UI Compose 한 Screen 의 modifier / state / handler 4건) 도 4 task 로 분리. wave-parallel 가치 손실 + 불필요한 wave 직렬화 + 4 commit fragmentation. 해결: **plan 작성 시점에** 메인이 직접 "같은 파일 + 테스트 경계 X + 모두 mechanical → 1 task 의 multi-step" 으로 묶도록 skill 본문에 룰 추가.

## 배경 / 문제 (dogfood finding)

v2.0.0 release 직후 사용자가 별도 외부 피처 (Android Compose chat 화면) 에서 `/auto-write-plan` 발화. 결과 plan task 들:

| Task | Files | 내용 |
|---|---|---|
| T1 | `AnthropicClient.kt` | Haiku helper `summarizeTitle()` 추가 |
| T2 | `ChatViewModel.kt` | `send()` 의 title 로직 교체 (T1 deps) |
| T3 | `MessageItem.kt` | `UserItem` 우측 정렬 + 색 |
| T4 | `Color.kt`, `STYLE_GUIDE.md` | lightness 상향 + 동기 |
| T5 | `ChatScreen.kt` | Scaffold contentWindowInsets + imePadding |
| T6 | `ChatScreen.kt` | 자동 스크롤 (`LaunchedEffect(messages.size)`) |
| T7 | `ChatScreen.kt` | BackHandler (drawer-우선) |
| T8 | `ChatScreen.kt` | 빈 화면 placeholder + 비서 톤 + 예시 prompt 3 |

**문제**:

- T5~T8 모두 `ChatScreen.kt` 한 파일만 만짐. file overlap 으로 같은 wave 진입 X → 4 wave 직렬.
- wave-parallel 가치 0 — `js-super-subagent-driven-development` 의 핵심 이점인 wall-clock 단축 손실.
- 4 commit fragmentation — `task 5 / task 6 / task 7 / task 8` 각자 commit. PR review 단계에서 squash 강요됨.
- TDD 분리 의미도 낮음 — UI 한 Screen 의 modifier / handler 추가는 mechanical 수준. 별도 테스트 경계 없음 (composable preview / 통합 UI 테스트는 한 번에 진행).

### Root cause

writing-plans skill 본문의 task 분해 가이드:

> Each step is one action (2-5 minutes)
> Tasks bigger than ~30 minutes → Decompose further

이 룰을 "단일 modifier 추가도 별도 task" 까지 적용 → over-decomposition.

진짜 의도: **TDD cycle (test → impl → pass → commit) 단위로 분해** — 즉 "테스트 경계가 다른 작업" 단위로 나누되, 같은 테스트 사이클에 묶일 수 있는 mechanical 변경은 1 task 안 multi-step 으로.

## 제안 — plan 작성 시점에 메인이 직접 묶기

### 채택안 — writing-plans / auto-writing-plans skill 본문 룰 추가

두 skill 의 task 분해 단계에 다음 룰 명시:

> **Same-file mechanical 묶음 룰 (v2.0.1+)**: 둘 이상의 logical change 가 다음 조건을 모두 만족하면 1 task 의 multi-step 으로 묶는다:
>
> 1. 모두 같은 파일 만짐 (Files 목록 동일)
> 2. 테스트 경계 없음 (한 통합 테스트로 검증 가능 또는 UI preview 수준)
> 3. 모두 mechanical (modifier 추가 / handler 등록 / Scaffold 옵션 등 — 알고리즘 변경 X)
>
> 이 경우 task 안 step 1 = test (한 번만), step 2~N = 각 변경의 byte-copy Edit, step N+1 = test pass, step N+2 = self-review.

핵심: **plan 작성 시점에** 메인 에이전트가 직접 판단 + multi-step 묶음 plan 으로 산출. 별도 Python helper / 자동 merge logic 추가 X — skill 본문 룰만 강화.

### 보조 — writing-plans Self-Review 단계 catch

writing-plans Self-Review 항목에 추가:

> Same-file mechanical 묶음 룰 위반 검사 — task 들 중 같은 파일만 만지는 chain 이 2건 이상 있는지 확인. 있으면 위 룰의 3 조건 재검토 → 묶을지 결정.

이 catch 는 메인이 룰 적용 깜박했을 때 안전망. auto 모드에서도 동일 catch (사용자 응답 wait X — 메인이 직접 결정).

### 후속 검토 — Python 자동 merge helper (v2.1+, 본 release 미포함)

후속 검토 가능한 idea: `scripts/dag_builder.py` 에 same-file-only chain 자동 검출 + 라인 영역 안 겹침 (`scripts/plan_byte_check.py` 의 line range 추출 재활용) → 자동 task collapse helper. 단:

- 자동 merge 가 잘못 catch 시 v2.0.0 byte-copy 룰 깨짐 → BLOCKED → reorder dispatch 비용 ↑
- skill 본문 룰 (위 채택안) 이 잘 작동하면 자동 helper 불필요

→ 본 release 채택안 운영 결과 보고 v2.1+ 후속 검토. 우선 v2.0.1 은 룰 단독.

## 좋은 점 (정량/정성)

### 정량

- **wave-parallel 효과 회복** — same-file 4 task → 1 task 면 4 wave 직렬 → 1 wave. wall-clock 75% 감소 추정 (4 wave 의 implementer + spec reviewer dispatch 비용 → 1 wave).
- **commit fragmentation 감소** — 4 commit → 1 commit. PR squash 작업 X.
- **plan 가독성 ↑** — 한 화면에 추가되는 4 변경이 한 task 안 step 으로 묶여 있어 "이 화면 변경" 의도 명확.

### 정성

- **TDD 본래 의도 회복** — 테스트 경계 단위로 분해. mechanical 변경은 한 cycle 안.
- **DAG 자동 build 효율** — file overlap 으로 deferred 되는 task 줄어듦. v2.0.0 의 dag_builder fix 와 시너지.

## 우려 사항 / 단점

### 1. mechanical 판정 모호성

writing-plans 의 메인 에이전트가 "mechanical 인지" 판단해야 함. 사용자가 동의 안 할 수 있음.

**Mitigation**:
- 룰 본문에 mechanical 예시 명시 (modifier 추가 / handler 등록 / Scaffold 옵션 / placeholder text 추가 등)
- 사용자가 plan 검토 단계에서 "분리해" 라고 catch 가능 (writing-plans Self-Review + 사용자 승인 단계)
- auto-flow 의 경우 사용자 catch 기회 부족 — 단, 본 룰은 보수적 default (애매하면 분해, mechanical 명확할 때만 묶음).

### 2. 1 task 가 너무 커지는 risk

같은 파일에 logical change 10건 묶이면 task 비대화.

**Mitigation**: 룰에 "최대 5건까지 merge" 같은 cap 추가 가능. 단 5+ 건이면 보통 mechanical 만으로는 끝나지 않음 (구조 변경 동반) → 자연 분리.

### 3. byte-copy 와의 정합성

v2.0.0 byte-copy 룰: implementer 가 plan 의 `**원본**` / `**수정 후**` 블록을 byte-identical Edit. multi-step task 에서 step N 이 step N-1 의 결과 위에서 또 Edit 한다면 — N-1 적용된 후의 파일 상태가 다음 step `**원본**` 이 되어야 byte-equal.

**Mitigation**: plan 작성 시 multi-step 의 각 `**원본**` 은 "직전 step 적용 후의 상태" 기준으로 작성. plan_byte_check helper 는 첫 step 만 검증 (이후 step 은 implementer 가 sequential 적용). 또는 helper 확장 — multi-step 의 각 `**원본**` 검증을 위해 plan 시뮬레이션 (복잡 — defer).

**현실적 해결**: multi-step task 의 step 들이 모두 file 의 다른 위치 (independent insertions) 면 안전 — 각 `**원본**` 은 원본 파일 기준 byte-equal. UI Compose modifier 추가 / handler 등록 등은 대부분 이 패턴.

### 4. spec-reviewer 검증 부담 ↑

multi-step task 1건 vs single-step task N 건 비교 시 spec-reviewer 가 한 번에 N 변경 검증 → review 깊이 ↓ 가능.

**Mitigation**: spec-reviewer prompt 에 "multi-step task 의 경우 step 별로 각각 검증" 명시. 단 같은 task 안 변경이라 cross-reference 자연.

### 5. RISK 분류 누락

multi-step task 안 한 step 만 RISK 트리거 (다른 step 은 trivial) 면 task 단위 RISK 분류가 over-conservative 가능.

**Mitigation**: wave finalization 의 메인 RISK 3-checklist 에 step 별 분리 가능. RISK 주석은 코드 라인에 직접 박아 정확.
