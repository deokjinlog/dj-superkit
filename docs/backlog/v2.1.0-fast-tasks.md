# v2.1.0 후보 — `fast-tasks` 신규 skill (다중 가벼운 task 병렬 흐름)

> 작성: 2026-05-16
> 우선순위: 중간 — brainstorming 풀 흐름이 무거운 케이스를 분리하는 별도 lightweight 경로.

## 문제

`brainstorming → designing-direction → writing-plans → executing-plans` 4단계 흐름은 **단일 피처를 깊이 있게 설계할 때** 유효. 그러나:

- 사용자에게 **할당된 task 10개를 한번에** 처리해야 하는 케이스가 잦음
- 그 10개가 **서로 종속도 낮음** (대부분 독립 또는 짧은 chain)
- 각각을 brainstorming 풀 흐름으로 돌리면 **너무 오래 걸림** (3-MD × 10 = 비현실적)
- 그렇다고 단일 brainstorming 으로 10개를 한 슬러그로 묶는 것도 어색 (피처 단위가 아님)

→ **가벼운 batch 모드**가 필요.

## 해결 — `fast-tasks` 신규 skill

### 입력 패턴

사용자가 한 turn 에 task 여러 개 (보통 5~15개) 통째로 던짐. 각 task 설명은 **거칠어도 OK** (정확도는 Socratic 으로 보강).

예시:
> "이번 스프린트 할당 10개 받았어. 1) 로그인 페이지 다크모드 / 2) /api/users GET endpoint 추가 / 3) Button 컴포넌트 size variant ... (계속)"

### Step 1 — Per-task Mini Socratic (메인 에이전트)

- 각 task **마다** 메인이 소크라테스식 질문 **2~3개** 던짐 (**3개 초과 금지**)
- 10개라면 → 라운드 10회 (task 1 질문 → 답 → task 2 질문 → 답 ...)
- 질문 목표: 각 task 의 명세 (입력/출력/제약) 를 최소한으로 명확화

### Step 2 — 단일 요구사항 MD 작성

산출물 1개:
```
docs/features/YYYY-MM-DD-fast-tasks-<batch-slug>/<batch-slug>-tasks.md
```

본문 구조:
```markdown
# <Batch slug> — Fast tasks (N개)

## Task 1: <title>
- 명세: <Socratic 합의된 내용>
- 영향 파일: <추정>

## Task 2: <title>
...

## Task N: <title>
...

---

## 병렬화 계획 (DAG)

- Chain A: 1 → 4 → 5   (이유: 4 가 1 산출물 의존, 5 가 4 산출물 의존)
- Chain B: 2 → 7       (이유: 7 이 2 export 사용)
- Independent: 3, 6, 8, 9, 10

총 5 wave-stream → 5 서브에이전트 병렬:
- subagent_1: 1 → 4 → 5
- subagent_2: 2 → 7
- subagent_3: 3
- subagent_4: 6
- subagent_5: 8 → (이어서) 9 → 10  (※ 또는 각각 독립이면 7 병렬)
```

### Step 3 — Smart Parallel Dispatch (메인 에이전트)

- 메인이 §병렬화 계획 읽기
- **Chain (순차 종속)**: 하나의 서브에이전트가 chain 순서대로 처리 (e.g. 1 끝나면 4, 4 끝나면 5)
- **Independent**: 각각 별도 서브에이전트로 동시 dispatch
- **극단 케이스**: 모두 독립이면 N 병렬
- 종속 판단: **LLM 휴리스틱** (메인이 §명세 + 영향 파일 보고 추론). 사용자 명시 X.

### 생략 항목 (lightweight 흐름)

| 항목 | 풀 흐름 (brainstorming) | fast-tasks |
|---|---|---|
| tech-design.md | ✅ | ❌ 생략 |
| implementation-plan.md | ✅ | ❌ 생략 |
| verifying-spec | ✅ | ❌ 생략 |
| 변경이력 footer | ✅ | ❌ 생략 |
| risk-annotation | ✅ | ❌ 생략 |
| docs-pretty / code-pretty | ✅ | ❌ 생략 |
| AskUserQuestion 게이트 | ✅ 다수 | ❌ Socratic 외 X |

→ 진짜로 단일 `<batch-slug>-tasks.md` **1 파일** + 실제 코드 변경 만 남음.

## Skill 발동 트리거

명시 invoke 만 (`/fast-tasks` slash command 또는 skill 직접 호출). brainstorming 의 자동 발동 경로와 충돌 X.

## Non-goals (이번 버전 안 함)

- 사용자가 명시 종속 그래프 그려주는 모드 (LLM 휴리스틱만)
- chain 내부 task 실패 시 자동 재시도 (사용자가 수동 재실행)
- 변경이력 / RISK 사후 보강 (이 흐름은 영구히 생략 — "가벼움"이 디자인 의도)

## 관련 결정

- **이름**: `fast-tasks` (대안 후보: `bulk-tasks`, `parallel-tasks`, `quick-batch`)
- **위치**: `skills/fast-tasks/SKILL.md`
- **slash command**: `commands/fast-tasks.md` (`/fast-tasks` 트리거)

## 결합 / 영향 범위

- brainstorming / designing-direction / writing-plans / executing-plans 본문 변경 0 (독립 경로)
- auto-* 본문 변경 0
- `js-super-subagent-driven-development` 의 wave-parallel 분담 로직 부분 재사용 가능 (chain/independent 분류 + dispatch 패턴) → 신규 skill 본문에 **mirror** 하되 본 skill 변경 X
- preflight.py 영향 0 (fast-tasks 는 별도 entry guard 가 필요하면 helper 추가, 기존 4-skill 변경 X)

## 미정 사항

- Socratic Q 갯수 cap (현재 안: per-task **2~3개, 3 초과 금지**) 정확한 수치 dogfood 후 조정
- batch size 상한 (10개? 20개? 너무 크면 메인 컨텍스트 burst)
- chain 길이 상한 (5 단계 넘는 chain 은 brainstorming 풀 흐름 권장으로 fallback?)
- DAG 분석 결과 노출 형식 (단순 리스트 vs Mermaid 다이어그램)
