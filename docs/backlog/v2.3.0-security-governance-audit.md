# v2.3.0 후보 — 전체 프로젝트 보안 & 거버넌스 감사 커맨드 (`/audit-risk`)

> 작성: 2026-05-21
> 우선순위: 중간 — 1회성 리포트, 코드 변경 X. 사용자 의사결정용.

## 문제

프로젝트 규모가 커지면 다음 위험이 산발적으로 누적되는데, 사용자가 한눈에 파악할 도구가 없음:

- **API 비용 폭주** — 외부 API (LLM / payment / SaaS) 를 무차별 호출하는 코드 (loop 안 호출 / 무한 retry / cache 미적용)
- **개인정보 노출** — log / response / 외부 송신 / 평문 저장 등에서 PII 누출
- **민감 로직 결함** — 사용량 카운터 / 비용 산정 / 결제 처리 에서 race / off-by-one / 검증 누락
- **에이전트 위험** — LLM 에이전트가 민감 데이터에 접근하거나 prompt 가 jailbreak 취약
- **거버넌스 회피** — 인증 우회 / 권한 검사 누락 / audit log 결여 / 보안 룰 wrap-around

→ 사용자 손으로 grep 하기엔 양 많고, 외부 보안 도구는 도메인 모름. **dj-superkit 가 프로젝트 컨텍스트 알고 있으니 LLM 으로 한 번 훑어주는 게 가장 빠름.**

## 해결 — `/audit-risk` 신규 커맨드

`commands/audit-risk.md` 1 파일. **커맨드만**, 스킬 X. 1회성 호출, 코드 수정 X, HTML 보고서 1개 생성.

### 흐름

**Step 1 — Subagent 5개 병렬 dispatch**

메인이 한 메시지에 5개 `Task` 호출 (Sonnet, `run_in_background: false` — 결과 wait):

| Subagent | 책임 | 핵심 동작 |
|---|---|---|
| **A — API Cost** | 외부 API 사용처 식별 + 과다 호출 패턴 + 비용 분석 | grep import / SDK / fetch / axios → 호출 빈도 추정 (loop / retry / cache) → `/context7` + WebSearch 로 각 API 단가 조회 → 월 추정 비용 |
| **B — PII Security** | 사용자 개인정보 처리 점검 | grep 개인정보 필드 (email / phone / ssn / 주민번호 / address / 카드번호) → log / response / 외부 송신 / DB 평문 저장 패턴 검출 → 마스킹·암호화 미적용 위치 리스트 |
| **C — Sensitive Logic** | 사용량 / 비용 / 결제 로직 결함 | grep `usage` / `cost` / `bill` / `payment` / `charge` / `subscription` / `quota` → race condition / off-by-one / 검증 누락 / 트랜잭션 미적용 / idempotency 부재 패턴 |
| **D — Agent Risk** | 에이전트 로직 / 프롬프트 위험 | LLM 에이전트 코드 검출 → 민감 데이터 접근 여부 → prompt injection 취약점 / system prompt 노출 / tool 호출 권한 과다 / output 검증 미적용. **에이전트가 민감부분 안 다루면 "PASS — 에이전트 위험 없음" 1줄 보고 후 종료** |
| **E — Governance** | 인증 / 권한 / audit log / 보안 룰 우회 | 인증 미들웨어 우회 라우트 / RBAC 미적용 endpoint / audit log 결여 / 환경변수 / secrets hardcoding / CORS 과다 허용 / SQL injection / XSS / CSRF 패턴 |

각 subagent 는 **자기 영역만 깊이 스캔** + 발견 항목별 (파일:라인 + 위험도 + 권장 조치) 구조화 리포트 반환.

**Step 2 — 메인이 결과 종합 + HTML 생성**

메인이 5개 subagent 결과 수신 → 또 다른 subagent (Sonnet) 1개 추가 dispatch:

- 입력: 5개 영역 raw 리포트 + 프로젝트 메타 (이름 / 설명 / 사용 스택)
- 출력: `docs/audit/<timestamp>-audit-risk.html` 단일 파일
- 룰: **자유 디자인 톤** (v2.2.4 generating-html 룰 mirror) — 사무 톤 금지, 인터랙티브 풍부, 별의별 UI 요소 OK
- 인터랙션: tab / accordion / filter / sort / search / sticky TOC / dark-mode toggle / 영역별 hero
- 시각화:
  - 위험도 분포 → donut / bar chart (inline SVG)
  - API 비용 → 카드 (월 추정 / 호출 빈도 / 비용 trend)
  - 발견 항목 → 카드 grid + 위험도 색깔 (red / orange / yellow / green)
  - 영역별 score → score bar / radar chart
  - 권장 조치 → 우선순위 큐 (Critical → Low)
- self-contained (inline CSS / SVG / JS, 외부 CDN X)
- 다크/라이트 모드 자동
- offline 렌더 (file:// double-click OK)

**Step 3 — 출력**

메인이 사용자에게 한 줄:
```
✅ 감사 완료: docs/audit/<timestamp>-audit-risk.html
- Critical: <N>건 / High: <N>건 / Medium: <N>건 / Low: <N>건
- 영역별 점수: API <X>/100, PII <X>/100, 민감로직 <X>/100, 에이전트 <X>/100, 거버넌스 <X>/100
```

## 영역별 상세 룰

### Subagent A — API Cost

**검출 대상**
- HTTP client (fetch / axios / got / requests / aiohttp 등) 사용처
- 외부 SDK (openai / anthropic / stripe / twilio / sendgrid / aws-sdk 등) 임포트
- 호출 패턴: loop 안 호출 / N+1 / retry 무한 / cache 미적용

**비용 분석**
- `/context7` 로 각 SDK 의 pricing 페이지 조회 (예: openai → tokens 단가)
- WebSearch fallback (context7 결과 부족 시)
- 호출 빈도 추정 (코드의 trigger 빈도 추정 — cron / user-action / event)
- 월 추정 비용 (best/worst case)

**위험도**
- Critical: 단가 ↑ + loop 안 호출 + cache 없음
- High: 단가 중 + retry 무한
- Medium: 단가 ↓ + 빈도 ↑
- Low: 단가 ↓ + 빈도 ↓

### Subagent B — PII Security

**검출 대상**
- 필드 keyword: email / phone / mobile / ssn / 주민(번호)? / passport / 카드(번호)? / cvv / address / 주소 / 생년월일 / birthday / dob / ip_address / device_id
- 평문 저장 (DB / file / localStorage / sessionStorage)
- log 출력 (`console.log` / `logger.info` / `print` / `slog` 등) 에 PII 직접 포함
- 외부 송신 (3rd party API body / GA / Sentry 등) 에 PII 포함
- response body 에 불필요 PII 포함

**위험도**
- Critical: log 또는 외부 송신에 PII 평문
- High: DB 평문 저장 + 암호화 X
- Medium: response 에 불필요 PII
- Low: 마스킹 안 된 internal 표시

### Subagent C — Sensitive Logic

**검출 대상**
- keyword: usage / cost / bill / payment / charge / subscription / credit / quota / limit / rate-limit
- 패턴:
  - **Race condition** — counter increment without lock / transaction
  - **Off-by-one** — `< limit` vs `<= limit` 혼용
  - **검증 누락** — 입력 sanitize / 음수 / overflow 미체크
  - **트랜잭션 X** — multi-step 결제 / 잔액 차감 + 부수 작업이 트랜잭션 밖
  - **Idempotency X** — 결제 / 외부 요청 재시도 시 중복 처리 위험

**위험도**
- Critical: race 또는 트랜잭션 X 가 비용 / 결제 로직에 발견
- High: idempotency X 가 결제 로직에
- Medium: off-by-one 또는 검증 누락
- Low: 잠재 위험 (실제 경로 X 인 dead code)

### Subagent D — Agent Risk

**Pre-check (시작 직전)**

프로젝트에 LLM 에이전트 코드가 있는지 grep:
- import 패턴: `openai` / `anthropic` / `langchain` / `crewai` / `autogen`
- system prompt / tool calling 코드
- agent loop / multi-turn

**없으면**: "PASS — 에이전트 위험 없음, 본 영역 점검 불필요" 1줄 반환 + 즉시 종료. (이 룰이 user 가 요청한 "에이전트가 민감부분을 다루지 않는다면 패스")

**있으면**: 다음 검출
- **민감 데이터 접근**: 에이전트 context / tool 가 PII / 결제 정보 / 인증 토큰에 접근하는지
- **Prompt injection 취약**: user input 이 system prompt 에 sanitize 없이 주입
- **System prompt 노출**: response 에 system prompt 그대로 노출 가능 패턴
- **Tool 호출 권한 과다**: 에이전트가 destructive 도구 (DB delete / payment / email send) 호출 가능 + 검증 X
- **Output 검증 미적용**: LLM output 을 그대로 코드 실행 / DB write / 외부 호출

### Subagent E — Governance

**검출 대상**
- **인증 미들웨어 우회**: 라우트 핸들러에 auth check 없음 (특히 admin / internal API)
- **RBAC 미적용**: 권한별 분기 없는 sensitive endpoint
- **Audit log 결여**: 민감 작업 (결제 / 권한 변경 / 데이터 삭제) 후 log 없음
- **Secrets hardcoding**: API key / DB password / JWT secret 이 코드에 평문
- **환경변수 부주의**: `process.env.X` 를 sanitize 없이 client 노출
- **CORS 과다 허용**: `*` 또는 너무 넓은 origin
- **SQL injection**: 문자열 concat 으로 query 조립
- **XSS**: HTML escape 미적용 사용자 입력 렌더링
- **CSRF**: state-changing endpoint 가 CSRF token 미검증
- **보안 룰 wrap-around**: `if (env === "dev")` 같은 분기로 prod 에서 검증 우회

**위험도**
- Critical: secrets hardcoded / SQL injection / 인증 우회 (admin)
- High: RBAC 누락 / XSS / CSRF
- Medium: audit log 결여 / CORS 과다
- Low: 잠재 위험 / dev-only 검증 우회

## 실행 모드 — 5 + 1 subagent

```
메인
 ├─ [parallel] A (API Cost)
 ├─ [parallel] B (PII)
 ├─ [parallel] C (Sensitive)
 ├─ [parallel] D (Agent — pre-check 후 PASS 가능)
 ├─ [parallel] E (Governance)
 │
 │   (5개 모두 await — 결과 수집)
 │
 └─ [sequential] F (HTML report generator, Sonnet)
     입력: A/B/C/D/E raw + 프로젝트 메타
     출력: docs/audit/<timestamp>-audit-risk.html
```

A~E 는 **read-only** — 코드 수정 X. F 는 HTML 1 파일 write.

## Non-goals (이번 버전 안 함)

- 자동 fix / 수정 PR 생성 (사용자 결정 후 별도 작업)
- 비용 예측 정확도 보증 (best-effort 추정)
- 보안 인증 / 컴플라이언스 (OWASP / SOC2 / GDPR 등) 표준 준수 증명
- 외부 보안 도구 (Snyk / SonarQube) 연동
- CI/CD 통합 (수동 호출만)
- 차분 감사 (이전 보고서와 diff) — 미래 버전 후보

## 영향 범위

- 신규: `commands/audit-risk.md`
- 신규: `docs/audit/` 디렉토리 + `.gitignore` 추가
- 기존 skill / command 영향 0

## 검증

- 빈 프로젝트 (소스 0) → 보고서 "위험 없음" 5 영역 모두 fixture
- 의도된 hardcoded secret 포함 fixture → Critical 검출
- 의도된 외부 API loop 호출 fixture → API Cost Critical
- LLM 에이전트 없는 프로젝트 → Subagent D "PASS" 1줄
- LLM 에이전트 있는 프로젝트 (민감 데이터 접근) → Subagent D 활성화 + 검출
- HTML self-contained 검증 (offline 렌더, 외부 URL 0)
- 위험도 분포 차트 / 점수 / 카드 grid 모두 렌더링 OK
