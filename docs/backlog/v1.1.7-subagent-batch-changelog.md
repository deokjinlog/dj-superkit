## 신고 본문 (3-layer 문제)

### 신고 1 (메인): "메인 에이전트가 끝에 한번에 정리해주길 원했는데 매 task마다 변경이력이 떴어"

#### 사용자가 원했던 워크플로우 (정확한 인용)

> 내가 원하는건 서브에이전트 일이 다끝나면 메인에이전트가 한꺼번에 계획서와 다른점 싹다 정리해주길 원했어 (서브에이전트 결과에 만약 코드 기존과 바뀐게 있으면 어디 바뀌었다라고 같이 메인에이전트에게 결과를 미리 남겨놓고 나중에 메인에이전트가 한번에 정리)

도식화:

```
[사용자가 원했던 흐름]

Per-task:
  Implementer subagent → "내가 무엇을 했고 plan과 무엇이 다른지" 메모 남기기 (in-memory or temp file)
  Spec reviewer → 합격/불합격
  Main agent → (변경이력 안 건드림, 기록만 누적)

모든 task 완료 후 1회:
  Main agent → 누적된 메모를 종합 → 변경이력 footer에 일괄 정리
                ↓
                * task 1~10 변화 통합 entry 1개 또는
                * task별 entry 일괄 append (한 commit에)
```

```
[v1.1.6 현재 구현 — 정반대]

Per-task:
  Implementer → 코드 변경 + commit
  Spec reviewer → 합격
  Main agent → 즉시 변경이력 [코드-수정] entry 1개 append + log commit

→ 10 task = 10번의 변경이력 append + 10번의 [log] commit
```

차이점: **누적 후 일괄** vs **즉시 매번**.

#### 왜 사용자가 그걸 원했나 (추론)

- Plan과 실제 구현의 갭(누락/초과)을 한눈에 보고 싶음
- 매 task entry는 task 안에서만 의미 — 끝에 봐야 전체 그림이 잡힘
- 마지막 종합 정리가 plan ↔ 코드 정합성 검증 + 다음 단계 (PR 작성, finishing-a-development-branch) 진입의 자연스러운 게이트

### 신고 2: 변경이력 entry와 git history의 정보 중복

각 task의 [코드-수정] entry는:
- 이유 / 무엇이 / 영향범위
- 변경 전 코드 / 변경 후 코드 (블록)
- 위험 카테고리

이 정보 대부분이 다음과 거의 동일:
- `git log --pretty=full` (왜 / 누가 / 언제)
- `git show <sha>` (변경 전/후)
- 본 commit message (`feat(...): ...` 의 이유)

`commit_policy: per-task` (git-fast 모드) 일 때는 git이 이미 audit trail. 변경이력에 변경 전/후 코드를 다시 옮겨 적는 건 **이중 저장소** — 하나만 있으면 충분.

**제안 방향**:
- git-fast 모드일 때 변경이력 entry는 **메타정보만** (id / 이유 / 영향범위 / 위험 카테고리 / 연관 commit SHA)
- 변경 전/후 코드는 `git show <sha>` 한 줄로 조회 가능 → entry에서 생략
- `commit_policy: single` / `none` 일 때만 변경 전/후 코드를 entry에 보존 (git이 못 추적하니까)

### 신고 3: schema에 비-코드 entry type 부재 (Task 10에서 노출)

v1.1.6 Task 10은 검증(grep) + git tag 작업만 했고 **코드 변경 0건**. 그런데 `[코드-수정]` 태그로 변경이력 entry가 추가됨 — tag/reality 불일치.

`change-history` 스킬의 entry type 목록:
- `[요구사항-수정]` / `[개발방향-수정]` / `[구현계획서-수정]` / `[코드-수정]` / `[API테스트]`

빠진 카테고리: **검증/릴리즈/태그만 있는 task**.

**제안 방향**:
- 신규 entry type `[검증]` 또는 `[릴리즈]` 추가
- 또는: schema에서 "코드 변경 0건" 케이스를 entry-skip으로 지정 (해당 task는 git tag/log만으로 추적)

## 사용자가 명시한 핵심 동작 (재확인)

서브에이전트 결과에 코드 변경 정보를 메인이 받기 쉽게 미리 남겨놓는 메커니즘이 필요:

> 서브에이전트 결과에 만약 코드 기존과 바뀐게 있으면 어디 바뀌었다라고 같이 메인에이전트에게 결과를 미리 남겨놓고 나중에 메인에이전트가 한번에 정리

이 부분이 핵심 인터페이스 변경. 지금 implementer 프롬프트는 "your job is to commit, main does governance"라고만 명시. "변경 위치/요약을 메인에 어떤 형식으로 패스할지"는 명시 안 됨 — implementer가 자유롭게 보고하고 메인이 알아서 git diff로 추출. 이걸 더 명확한 인터페이스로 바꿔야 batch processing이 가능.

후보 인터페이스 (brainstorming에서 결정):

| 옵션 | 설명 |
|---|---|
| A | Implementer가 임시 파일 (예: `.changelog-buffer/task-N.md`) 에 변경 요약 적기. Main이 모두 끝나면 합쳐서 변경이력 footer에 1번에 append, 임시 파일 삭제. |
| B | Implementer 보고 형식을 더 구조화 (status / files-changed / commit-sha / risk-hints). Main은 메모리에 누적, 끝에 일괄 처리. |
| C | git notes 사용 — 각 commit에 메타 정보 attach. Main이 끝에 git notes 리스트를 모아 변경이력 정리. |
| D | 아무 변경 없이 — main이 끝에 `git log --since=BASE_SHA` 만 보고 직접 정리 (현재 인프라로 가능, implementer 변경 불필요). |

각 옵션의 trade-off는 brainstorming/designing-direction 단계에서 다룰 것.

## 수용 기준 후보 (brainstorming에서 다듬기)

- AC-1: 서브에이전트 모드로 N개 task 실행 후, 변경이력 footer에 누적된 [코드-수정] entry는 N개 (or 1개 종합 entry, 옵션에 따라). 매 task 시점에는 추가 안 됨.
- AC-2: 종합 정리 시 plan과 실제 코드의 차이점이 하나의 메시지로 main agent를 통해 사용자에게 노출됨.
- AC-3: 코드 변경 0건인 task (검증/태그만)는 [코드-수정] entry로 기록되지 않음. 별도 entry type 또는 entry skip.
- AC-4: git-fast 모드일 때 변경이력 entry는 변경 전/후 코드 블록을 생략해도 되고, git show로 즉시 조회 가능.
