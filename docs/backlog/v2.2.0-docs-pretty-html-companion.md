# v2.2.0 후보 — docs-pretty .html 동봉본 (사람용 가독성)

> 작성: 2026-05-16
> 우선순위: 중간 — md 만으로는 사람이 표·구조·관계를 빠르게 파악하기 어려운 경우 보강.

## 문제

dj-superkit 흐름의 3-MD (`<slug>-requirements.md` / `<slug>-tech-design.md` / `<slug>-implementation-plan.md`) 는 **AI 가 읽기 좋은 포맷**으로 작성됨. 표·헤더·코드블록 활용도 docs-pretty 가 잡아주지만:

- 사람이 긴 MD 를 stream 으로 훑을 때 **구조 파악 + 관계 시각화** 가 약함
- 다이어그램 / 풍부한 표 / 시각 hierarchy 필요한 정보 (DAG, 데이터 흐름, 컴포넌트 관계) 가 텍스트로만 노출
- 결과: 사용자가 docs-pretty 처리된 MD 를 받아도 **빠른 한눈 이해가 어려움**

## 해결 — `.html` 동봉본 (사람용 가독성 사본)

### 핵심 패턴

docs-pretty 가 발동될 때 **두 subagent 를 병렬로 dispatch**:

| Subagent | 모델 | 산출물 | 대상 독자 |
|---|---|---|---|
| A (기존) | Sonnet | `.md` (format-only pass) | AI + 사람 |
| B (신규) | Sonnet | `.html` (시각화 풍부 사본) | **사람만** |

**AI 는 항상 `.md` 만 읽음.** `.html` 은 사람이 브라우저로 열어서 빠르게 훑는 용도. brainstorming / designing-direction / writing-plans / executing-plans 의 모든 AI 처리 흐름은 `.md` 기준 그대로.

### `.html` 디자인 룰

1. **의미 보존 절대** — `.md` 의 문장 / 항목 / 결정 사항 / 숫자 / 표 내용 1:1 매핑. 의미 누락 X / 의역 X.
2. **시각화 풍부** —
   - 헤더 hierarchy 색·여백·typography 로 명확하게
   - 표 → 정렬·줄무늬·sticky header
   - DAG / 데이터 흐름 / 컴포넌트 관계 → Mermaid 다이어그램 또는 inline SVG
   - 코드 블록 → syntax highlight
   - 항목 간 관계 (e.g. CH-id → 영향범위) → cross-link 또는 anchor
3. **인쇄·offline 가능** — 외부 의존 0 (CDN X). inline CSS + inline SVG / Mermaid 도 self-contained 빌드.
4. **다크/라이트 모드 자동** — prefers-color-scheme 활용.

### 산출물 위치

기존 `.md` 와 같은 폴더, 같은 basename, 다른 확장자:

```
docs/features/2026-05-16-foo/
├── foo-requirements.md      ← AI + 사람
├── foo-requirements.html    ← 사람 전용 (신규)
├── foo-tech-design.md
├── foo-tech-design.html
├── foo-implementation-plan.md
└── foo-implementation-plan.html
```

### 동기화 boundary (docs-pretty 와 동일)

- **초기 작성 iteration loop** 에서만 발동 — `.md` 와 `.html` 둘 다 매 draft 마다 재생성
- **첫 change-history entry 가 logging 되는 순간 STOP** — live doc 진입. 이후 부분 수정 시 `.html` 자동 재생성 X
- live doc 진입 후 `.html` 이 stale 되어도 무시 — `.md` 가 single source of truth. 사용자가 원하면 명시 invoke 로 재생성

### 병렬 dispatch 명세

docs-pretty SKILL.md 의 Procedure 섹션 수정:

```markdown
**Step 5 — Dispatch parallel subagents:**

Run two Task tool calls in a single message (parallel):
- Subagent A: Sonnet, format-only pass on .md (existing behavior)
- Subagent B: Sonnet, generate .html companion from the prettified .md output

Subagent B receives:
- The prettified .md (from Subagent A's expected output, or RAW .md if A hasn't finished — main reconciles)
- Strict prompt: "preserve meaning byte-equivalently at the semantic level, enhance visual layout only"

Main agent waits for both to complete, then validates:
- .md change-history footer untouched (A's contract)
- .html exists at sibling path (B's contract)
- .html basename matches .md basename
```

### 충돌 / 실패 모드

| 상황 | 처리 |
|---|---|
| A 성공 / B 실패 | `.md` 만 갱신 + 사용자에게 ".html 생성 실패, 수동 retry 가능" 알림 |
| A 실패 / B 성공 | 둘 다 폐기 (A 가 핵심 — `.md` 없으면 의미 X) |
| A·B 둘 다 실패 | docs-pretty 자체 abort, RAW `.md` 유지 |
| `.html` semantic drift 의심 | B 결과 메인이 spot-check (제목·항목 수 매칭) — drift 발견 시 폐기 |

### docs-pretty 의 기존 룰 보존 (변경 X)

- 초기 작성 iteration loop 에서만 발동
- 첫 change-history entry 후 STOP
- Sonnet 으로 dispatch
- 사용자 fix iteration 마다 재발화
- pre-flight (target 존재 + footer 비어있음 + filename 패턴 매칭)

### Anti-patterns

- `.html` 생성 시 `.md` 본문 의역 / 요약 / 재구조화 → **금지**. 의미 보존 1:1
- AI 가 `.html` 읽어서 작업 → **금지**. 모든 AI 흐름은 `.md` 만
- 사용자 fix turn 마다 `.html` 재생성 비용 부담 시 → 매 iteration 마다 둘 다 갱신 (caching X — Sonnet 비용 감수)
- `.html` 외부 CDN 의존 (jQuery / 외부 CSS) → 금지, self-contained

## 결합 / 영향 범위

- **`docs-pretty` SKILL.md 본문 수정** — Procedure 섹션에 parallel dispatch + Subagent B 명세 추가
- **`code-pretty` 영향 0** — 별도 skill, 본 변경과 독립
- **brainstorming / designing-direction / writing-plans / executing-plans 영향 0** — `.md` 만 읽는 흐름 그대로
- **change-history 영향 0** — `.html` 은 change-history 트래킹 대상 X (live doc 진입 후 stale 허용)
- **`.gitignore` 결정 필요** — `.html` 동봉본 commit 할지 (사람 리뷰용 cross-machine 공유) vs gitignore (AI 가 쓰지 않으므로 노이즈) — **미정**
- **auto-* / og-* 영향 0**

## Non-goals (이번 버전 안 함)

- 실시간 `.html` 미리보기 서버 / VSCode preview 통합
- `.html` 다국어 (영문/한글 자동 분기) — `.md` 가 그대로 한국어면 `.html` 도 그대로
- `.html` 인터랙티브 (탭/접힘/검색) — 정적 페이지만
- `.html` ↔ `.md` 양방향 sync (`.html` 수정 → `.md` 반영) — `.md` 만 source

## 검증

- `.md` 와 `.html` basename 1:1 매칭 fixture
- `.md` 본문 문장 수 vs `.html` 의 문장-level node 수 차이 5% 이내 (의미 보존 spot-check)
- offline 검증: `.html` 더블클릭으로 열어 표·다이어그램 렌더링 OK
- 다크모드 자동 전환 OK
- live doc 진입 후 추가 docs-pretty 발동 X (기존 boundary 보존)

## 미정 사항

- Subagent B 의 모델 (Sonnet vs Haiku — 시각화 풍부할수록 Sonnet 권장)
- `.html` gitignore 여부
- Mermaid 렌더링 전략 (inline JS 임베드 vs 미리 SVG 로 컴파일)
- 사용자 fix 1턴마다 `.html` 재생성 비용 — 너무 비싸면 명시 invoke 시만 `.html` 갱신하는 옵션 분기
- Subagent B 가 처리할 시각화 휴리스틱 (어느 표가 다이어그램으로 가야 하는지 판단) — 명시 룰 vs LLM 자율

## 비용

- docs-pretty 호출당 Sonnet 호출 **2회** (기존 1회 → 2회)
- 초기 작성 iteration N 회 × 2 subagent — Sonnet × 2N 호출
- 사용자 fix 마다 재발동 → 추가 비용 누적
- mitigation: 미정 사항의 "fix 마다 갱신 vs 명시 invoke" 옵션 분기로 흡수 가능
